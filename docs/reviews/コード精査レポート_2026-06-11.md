## 🔴 C-1. `ALLOWED_HOSTS` がハードコードで上書きされ、環境変数が無効→ ✅ 対応済み（2026-06-11）

- **箇所**: `shift_manager/settings.py:41`

```python
ALLOWED_HOSTS = (
    [h.strip() for h in os.environ.get('DJANGO_ALLOWED_HOSTS', '').split(',') if h.strip()]
    if not DEBUG
    else []
)
ALLOWED_HOSTS = ('shiftmanager.info', 'localhost', '192.168.10.8') if not DEBUG else []  # ← この行が直前の行を完全に上書き
```

- **内容**:
  1. 36〜40行目（環境変数 `DJANGO_ALLOWED_HOSTS` を読む処理）が**デッドコード**になっている。systemd の `Environment=DJANGO_ALLOWED_HOSTS=...` も、compose の `.env` の `ALLOWED_HOSTS` も**一切効いていない**。
  2. 運用マニュアル・lecture 13/15/19章・`.env.example` はすべて「`DJANGO_ALLOWED_HOSTS` で設定する」と説明しており、**ドキュメントと実態が食い違う**。別ホストへ展開した読者は、環境変数を正しく設定しても `400 DisallowedHost` になる（許可されるのはハードコードの3つだけ）。
  3. 公開ドメイン `shiftmanager.info` と LAN の内部IP `192.168.10.8` が**リポジトリに直書き**されている（環境情報の漏洩・環境依存値の混入。セキュリティ精査 A-0 の「dev/prod 混在」の典型例）。
- **経緯の推測**: 公開作業時のデバッグで一時的に足した行がそのまま残ったとみられる。50行目のコメントアウト（`#CSRF_TRUSTED_ORIGINS += [...]`）も同種の痕跡。
- **推奨**: 41行目を削除して環境変数方式に戻す。**ただし削除前に**、現在動いている systemd サービス定義の `DJANGO_ALLOWED_HOSTS` に `shiftmanager.info`（と必要なら LAN IP）が入っているか確認すること。**今の本番はこのハードコードのおかげで動いている**ため、行を消すだけだと本番が 400 になる恐れがある。50行目のコメントアウト行も合わせて削除してよい。

### C-2. フォームセットの隠しフィールド改ざんで 500 エラー／不正データ → ✅ 対応済み（2026-06-11）

- **箇所**: `shifts/forms.py`（`ShiftDayForm.work_date`、`FixedShiftDayForm.weekday` はいずれも `HiddenInput`）、`shifts/views.py` の `_save_submission` / `_save_fixed_shift` / `_apply_payload`
- **内容**: 隠しフィールドはサーバー側で値域・重複を検証していないため、POST を細工すると:
  - **重複した `work_date` / `weekday`** → `bulk_create` がユニーク制約（`unique_day_per_request` / `unique_fixed_shift_per_weekday`）に当たり **IntegrityError → 500**。
  - **期間外の `work_date`** → そのまま保存される（セキュリティ精査 L-2 で既知）。提出状況一覧は期間内の日付しか表示しないので実害は小さいが、データとしては不正。
  - **範囲外の `weekday`（例: 99）** → そのまま保存される（DBはchoicesを強制しない）。さらに固定シフトの**変更申請の payload に細工があると、リーダーが承認操作をした瞬間に 500**になる（被害者が本人でなくリーダーになる点が嫌らしい）。
- **補足**: フォーム数（`TOTAL_FORMS`）の上限は Django の formset 既定（max_num=1000、絶対上限2000）で一応頭打ちになるため、無制限ではない（セキュリティ精査 L-2 の表現を補正）。
- **推奨**: ビューまたはフォームで「`work_date` が期間内・重複なし」「`weekday` が 0〜6・重複なし」を検証し、違反は 500 ではなくフォームエラーにする。ログイン済みユーザーしか送れないため緊急度は中。
- **対応（2026-06-11）**: `shifts/forms.py` に `BaseShiftDayFormSet`（`valid_dates` を受け取り、期間外・重複の `work_date` を検証）と `BaseFixedShiftFormSet`（範囲外・重複の `weekday` を検証）を追加。`shift_submit` ビューは `valid_dates=dates` を渡す。承認経路（`_apply_payload`）は、万一不正な payload が紛れてもリーダーの承認が落ちないよう重複・範囲外を無視する防御を追加。テスト4件追加（`shifts/tests.py`）。

### C-3. 締切超過後の「募集再開」が、ポリシーによっては何も再開しない

- **箇所**: `shifts/views.py` の `manage_period_close`（手動トグル）と `_submission_permission`
- **内容**: 締切日時を過ぎた期間を手動で「募集中に戻す」と、画面には「募集中に戻しました」と出るが、提出可否は `post_deadline_policy` の判定に戻るだけ。ポリシーが「提出・編集とも不可（lock_all）」の期間では、**再開してもクルーは誰も提出できない**。「未提出者のみ提出可」でも編集は再開されない。本当に全面再開したい場合は締切日時自体を未来に直すしかなく、UIの文言と実挙動がずれている。
- **推奨**: 仕様としてどちらが正か決める。「再開＝締切前と同じ状態に戻す」なら判定にフラグを足す、「再開＝ポリシー判定に戻すだけ」なら成功メッセージにその旨を補足する。

### C-4. ドキュメントと実装の権限差: 固定シフト編集許可の切替はリーダー不可

- **箇所**: `accounts/views.py` の `account_toggle_fixed_edit` / `account_bulk_fixed_edit`（いずれも `@admin_required`）と [features/fixed-weekly-shift.md](../features/fixed-weekly-shift.md)
- **内容**: 機能ドキュメント 2-2 は「クルー本人が編集してよいかは**リーダーが切り替える**」とあるが、実装はアカウント管理画面内のため**システム管理者（admin）のみ**が切り替え可能。リーダーには切替手段がない（同ドキュメント4章の実装メモは `@admin_required` と書いており、文書内でも記述が割れている）。
- **推奨**: 運用上リーダーに切らせたいなら `@manager_required` の専用エンドポイントへ移す。現状の admin 限定でよいなら、ドキュメント 2-2 の記述を直す（docs 側の修正は注記済み）。

### C-5. 管理者が自分のパスワードを「再発行」すると即ログアウトする → ✅ 対応済み（2026-06-11）

- **箇所**: `accounts/views.py:69` `account_reset_password`
- **内容**: 対象が自分自身でも実行でき、`set_password` でセッション認証ハッシュが変わるため、**実行直後に自分がログアウト**される。新パスワードは直前のページにメッセージ表示される設計だが、ログアウト後のログイン画面ではメッセージが見えない可能性があり、表示を見逃すと自分のパスワードが分からなくなる（他の管理者に再発行してもらうしかない）。
- **推奨**: 自分自身への再発行をブロックする（自己無効化・自己削除の防止と同じ発想）か、`update_session_auth_hash` で継続させる。
- **対応（2026-06-11）**: `account_reset_password` で対象が自分自身なら再発行せず、「設定画面のパスワード変更を使うように」案内して戻す（自己無効化・自己削除の防止と同じ発想）。テスト2件追加（`accounts/tests.py`）。
