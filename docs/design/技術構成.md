> 仕様書（`仕様書.md` v0.2）・DB設計（`データベース設計.md`）・画面設計（`画面設計.md`）に基づく技術構成。

## 1. 採用技術と選定理由

| レイヤ | 採用 | 理由 |
| --- | --- | --- |
| 言語 | **Python** | 既存の経験を活かせる。学習資料が豊富 |
| Webフレームワーク | **Django** | 認証・ユーザー管理・**管理画面**・ORMが標準装備。今回の機能と相性が良い |
| データベース | **SQLite（開発）/ MySQL（本番）** | 開発は手軽に、本番は同時アクセス・信頼性を確保 |
| テンプレート/画面 | **Django Templates + 少量のJavaScript** | 1フレームワークで完結。必要箇所のみJSを足す |
| CSS | **Bootstrap（または素のCSS）** | スマホ対応のレスポンシブを手早く |
| 本番アプリ起動 | **Gunicorn（WSGI）** | Djangoの定番アプリサーバー |
| リバースプロキシ | **Nginx** | 静的ファイル配信・HTTPS終端・プロキシ |
| HTTPS | **Let's Encrypt（certbot）** | 無料のTLS証明書 |
| バージョン管理 | **Git** | 変更管理（リポジトリは後で用意） |

### なぜフルスタックDjango（SPAにしない）か
- スキルアップ目的でも、まずWebの基礎（ルーティング・ORM・認証・テンプレート）を一通り通すのが効率的。
- 提出画面の時刻入力などインタラクティブな箇所にだけJavaScriptを足していけば、段階的にフロント技術も学べる。
- 認証・権限・管理画面をDjangoに任せられるため、本質的な機能に集中できる。

## 2. Django プロジェクト構成（案）

```
shift_manager/                # プロジェクトルート
├── manage.py
├── requirements.txt          # 依存パッケージ
├── shift_manager/            # 設定パッケージ
│   ├── settings.py           # 設定（開発/本番で分離も可）
│   ├── urls.py
│   └── wsgi.py
├── accounts/                 # ユーザー・認証アプリ
│   ├── models.py             # カスタムUser（role, email任意 など）
│   ├── views.py
│   └── ...
├── shifts/                   # シフト関連アプリ
│   ├── models.py             # ShiftPeriod, ShiftRequest, ShiftRequestDay, ...
│   ├── views.py              # 提出・一覧・確定シフト
│   ├── forms.py              # 入力フォーム＋バリデーション
│   └── templates/shifts/
├── templates/                # 共通テンプレート（base.html 等）
├── static/                   # CSS/JS/画像
└── media/                    # アップロードされた確定シフトPDF
```

### モデルとDB設計の対応

| DB設計のテーブル | Djangoモデル（配置） |
| --- | --- |
| users | `accounts.User`（`AbstractUser` を拡張し role/email を持つ） |
| shift_periods | `shifts.ShiftPeriod` |
| shift_requests | `shifts.ShiftRequest` |
| shift_request_days | `shifts.ShiftRequestDay` |
| shift_request_histories | `shifts.ShiftRequestHistory`（snapshot=JSONField） |
| confirmed_shifts | `shifts.ConfirmedShift`（FileField） |

> 注: Djangoの認証はカスタムUserモデルを**最初に**設定するのが定石（後からの変更は手間）。プロジェクト初期に `AUTH_USER_MODEL` を決める。

## 3. 画面とDjangoの対応

| 画面ID | 実装方針 |
| --- | --- |
| S1 ログイン | Django標準の認証ビュー（`LoginView`）を利用 |
| S2 ホーム | 自作ビュー。ロールで表示を出し分け |
| S3 シフト提出 | 自作フォーム（formset で各日入力）＋バリデーション＋JSで時刻入力補助 |
| S4 提出状況一覧 | 自作ビュー。LEFT JOIN相当で未提出者を赤表示 |
| S5 確定シフト閲覧 | FileField のPDFを表示/ダウンロード。アップロードはリーダーのみ |
| S6 アカウント管理 | **自作画面**。`UserCreationForm` 相当＋ModelFormで作成/編集、パスワードリセット、有効/無効化 |
| S7 期間・締切設定 | **自作画面**。ShiftPeriod の ModelForm で作成/編集、手動締切、削除 |
| S8 リーダー用トップ | 自作ビュー。管理機能への導線をまとめたメニュー |

- 管理系（S6/S7/S8）はアプリ内の専用画面として実装する。`/admin` はスーパーユーザー用の予備として残す。
- リーダー専用画面は、ログイン必須＋ロール判定（`is_leader`）でアクセス制御する。

## 4. 開発環境

- Python 3.x（仮想環境 `venv` を利用）
- `pip install django` ほか（`requirements.txt` で管理）
- 開発サーバー: `python manage.py runserver`
- DB: 開発はSQLite（設定不要）
- エディタ: VS Code（Python拡張）

### 想定する初期セットアップの流れ（概要）
1. 仮想環境作成・Djangoインストール
2. プロジェクト/アプリ作成（`accounts`, `shifts`）
3. カスタムUserモデルを定義し `AUTH_USER_MODEL` を設定
4. 各モデルを定義 → `makemigrations` / `migrate`
5. 管理画面にモデルを登録（S6/S7用）
6. 画面（S1〜S5）を順に実装

## 5. 本番運用（自宅 / 大学サーバー）

```
[ブラウザ] --HTTPS--> [Nginx] --proxy--> [Gunicorn] --> [Django] --> [MySQL]
                          |
                          +--> 静的ファイル/メディア(PDF) を配信
```

- **OS**: Linux（サーバーに準拠）
- **アプリ**: Gunicorn で Django を起動（systemd サービス化して常駐）
- **プロキシ**: Nginx がリクエストをGunicornへ転送、静的/メディアファイルを配信
- **DB**: MySQL（同一サーバー or 学内DB）。Djangoからは `mysqlclient` ドライバで接続
- **HTTPS**: ドメインが用意できれば Let's Encrypt。学内のみなら学内CA/自己署名も選択肢
- **設定の分離**: `DEBUG=False`、`ALLOWED_HOSTS`、`SECRET_KEY` は環境変数で管理
- **バックアップ**: DBの定期ダンプ、media（PDF）のバックアップ

### 確認が必要なサーバー側の条件
- 外部（学外/インターネット）からアクセスさせるか、学内/LAN内のみか
- 固定IP / ドメイン名の有無、開放できるポート（80/443）
- サーバーOSと管理権限（sudo可否）、PostgreSQLの利用可否

---

## 6. セキュリティ要件（最低限）

- パスワードはハッシュ保存（Django標準のハッシュ機構＝PBKDF2等。argon2も選択可）
- 通信はHTTPS化
- CSRF対策（Django標準で有効）
- ロールに応じたアクセス制御（リーダー専用画面の保護）
- `DEBUG=False` で本番運用、秘密情報はソースに直書きしない
