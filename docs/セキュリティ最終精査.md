# セキュリティ・致命バグ 最終精査レポート

- 対象: `accounts/` `shifts/` `shift_manager/` `templates/` `deploy/` および設定一式
- 方法: 認証・権限・IDOR・CSRF・XSS・ファイルアップロード/配信・秘密情報・SQL・データ整合の観点でコードを通読＋横断検索
- 方針: **本レポートは指摘のみ。修正は次セッションで行う**（このセッションではコード編集しない）
- 重大度: 🔴高（早めに対処）／🟡中（対処推奨）／🟢低（任意・将来）

> まず安心材料：SQLは全てORM経由で**SQLインジェクションの懸念なし**、テンプレートはDjangoの自動エスケープが効き`|safe`は組込のパスワードヘルプ1箇所のみで**XSSリスク低**、CSRFミドルウェア有効で全フォームに`{% csrf_token %}`、各ビューに権限デコレータが網羅、IDORは`request.user`紐付けと所有者チェックで防御、自己ロックアウト防止あり、保存処理は`transaction.atomic`。**致命的に機能を壊すバグは検出されませんでした。** 以下は主に「公開運用時の堅牢化」が中心です。

## 🔴 高

### H-1. 確定シフトPDF（media）が無認証で配信される（アクセス制御不備）
- **箇所**: `shifts/templates/shifts/confirmed.html:21`（`{{ cs.file.url }}` への直リンク）、`shift_manager/settings.py` の `MEDIA_URL`、本番は Nginx/Cloudflare が `/media/` を直接配信。
- **内容**: 確定シフトの閲覧はログイン必須だが、**ファイル本体のURL（`/media/confirmed_shifts/period_<id>/<名前>`）はログイン無しで誰でもダウンロード可能**。URLはほぼ推測可能（期間ID＋ファイル名）。シフト表という個人情報・勤務情報が外部流出しうる。
- **推奨**: メディアを**認証付きビュー経由**で配信する（Djangoビューで`login_required`＋`FileResponse`、または Nginx の `X-Accel-Redirect` / `internal` location で「認証OKのときだけ実ファイルを返す」）。少なくとも公開（Cloudflare）時は `/media/` を直アクセス不可にする。

### H-2. `DEBUG` の既定値が `True`（本番事故の地雷）
- **箇所**: `shift_manager/settings.py`：`DEBUG = os.environ.get('DJANGO_DEBUG', '1') != '0'`。
- **内容**: 環境変数 `DJANGO_DEBUG=0` を**設定し忘れると本番でも `DEBUG=True`**。例外時にソース・設定・環境変数が画面に出る重大な情報漏洩になり、`ALLOWED_HOSTS` も無効化される。今の運用はsystemdで`0`を渡しているが、別環境への展開時に忘れやすい。
- **推奨**: 既定を「安全側」に倒す。例: `DEBUG` 既定 `False`、開発時のみ `DJANGO_DEBUG=1` を明示。または起動時に「DEBUG=True かつ ALLOWED_HOSTSに公開ホスト」を検知して警告/拒否。`manage.py check --deploy` を公開前手順に明記（13章済）。

## 🟡 中

### M-1. ログインのブルートフォース対策が無い
- **箇所**: `accounts/urls.py`（Django標準 `LoginView`）。
- **内容**: 試行回数制限・アカウントロック・遅延・CAPTCHAが無く、辞書/総当たり攻撃に弱い。公開（Cloudflare）後は特に。
- **推奨**: `django-axes` 等で失敗回数制限、もしくは Cloudflare 側のレート制限/WAF・Cloudflare Access（メール認証）でログイン画面を保護。

### M-2. アップロードPDFの内容検証が拡張子・サイズのみ
- **箇所**: `shifts/forms.py` `ConfirmedShiftForm.clean_file`（`.pdf` 末尾一致＋10MB のみ）。
- **内容**: 実体がPDFかの検証（マジックバイト/MIME）が無く、任意ファイルを `.pdf` 名で保存可能。配信時に `Content-Disposition` 指定が無いとブラウザでインライン表示され、細工ファイルの悪用余地が残る。
- **推奨**: 先頭バイト（`%PDF-`）やMIME検証を追加。配信は `Content-Disposition: attachment`（ダウンロード強制）に。アップロードは管理権限者のみなので緊急度は中。

### M-3. HTTPS向けセキュア設定が既定で無効
- **箇所**: `shift_manager/settings.py`（`SECURE_PROXY_SSL_HEADER`/`*_COOKIE_SECURE` は環境変数指定時のみ。`SECURE_HSTS_*`・`SECURE_SSL_REDIRECT` は未設定）。
- **内容**: 公開時に有効化を忘れると、Cookie がHTTPでも送出されうる・HSTS無し等。`check --deploy` でも警告が出る項目。
- **推奨**: 公開構成では `DJANGO_SECURE_COOKIES=1`＋`DJANGO_BEHIND_TLS_PROXY=1` を必須化（17章で手順化済）。将来 HSTS も検討。

### M-4. `SECRET_KEY` 未設定時のフォールバックが危険側
- **箇所**: `shift_manager/settings.py`：`SECRET_KEY = os.environ.get('DJANGO_SECRET_KEY', get_random_secret_key())`。
- **内容**: 未設定だと**プロセス毎に別鍵**→複数ワーカーで「勝手にログアウト」、再起動で全セッション失効。今回実際に踏んだ問題（13章で固定鍵設定により回避済みだが、設定漏れで再発しうる）。
- **推奨**: 本番（`DEBUG=False`）で `DJANGO_SECRET_KEY` 未設定なら**起動時にエラーで停止**させる（フェイルファスト）。開発時のみフォールバック許可。

## 🟢 低

- **L-1. `confirmed_shift` が `is_visible` を未チェック**（`shifts/views.py:264`）。非表示期間の確定シフトもURL（pk）で閲覧可能。設計上「全員閲覧」だが、非表示期間は隠す方が一貫。
- **L-2. 提出の `work_date` を期間範囲で検証しない**（`shifts/views.py:169`、`_save_submission`）。フォームから期間外の日付を送れる。データ整合の軽微な問題。`formset` の `TOTAL_FORMS` も無制限のため、極端に多いフォーム送信で負荷をかけられる（`max_num`/件数チェック推奨）。
- **L-3. `/admin/` が露出**（`shift_manager/urls.py:8`）。superuserのみだが攻撃面。固定パスの変更や Cloudflare Access での保護が可能。
- **L-4. 依存バージョンがレンジ指定**（`requirements.txt`）。完全再現には `pip freeze > requirements.lock` の併用を推奨。
- **L-5. 初期/再発行パスワードを画面メッセージに平文表示**（`accounts/views.py:32,79`）。仕様上「一度だけ表示」だが、共有PC・肩越し・ブラウザ履歴に注意（運用上の留意）。
- **L-6. `password_change.html:31` の `|safe`**：Django組込ヘルプ文のため実害は低いが、`|safe` 使用箇所として記録。

## 致命的バグ（機能破壊）について

- 通読の範囲で**機能を壊す致命バグは未検出**。
- 権限・IDOR・自己ロックアウト防止・トランザクション・PROTECT＋削除前チェックは適切に実装されている。
- 強いて挙げれば L-2（work_date範囲・フォーム数）程度のデータ整合の緩さ。

---

## 良い実装（維持したい点）

- ORM徹底でSQLi無し／自動エスケープでXSS低／CSRF全面有効。
- ビューの権限デコレータ網羅、所有者チェックでIDOR防御。
- 自己の無効化・降格・削除の防止（最低1管理者を担保）。
- `ShiftPeriod.created_by`/`ConfirmedShift.uploaded_by` は `PROTECT`＋削除前チェックで整合維持。
- 秘密情報はコード/Gitに直書きせず環境変数・systemd側（[10章方針]）。

## 次セッションでの対応の優先順位（提案）

1. **H-1**（media認証配信）と **H-2**（DEBUG既定の安全化）— 公開運用の前提として最優先。
2. **M-4**（SECRET_KEY未設定で停止）・**M-3**（公開時セキュア設定の必須化）— 設定漏れ事故の予防。
3. **M-1**（ログイン保護）・**M-2**（アップロード検証＋添付配信）。
4. **L-1/L-2**（可視性・入力検証）ほかは余力で。

> いずれも対応時は `accounts/tests.py` `shifts/tests.py` にケースを追加し、`manage.py test` と `manage.py check --deploy` で回帰確認するのが安全です。
