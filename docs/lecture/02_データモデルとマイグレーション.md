# 2章 データモデルとマイグレーション

0章で設計したデータを、Djangoの**モデル**として書きます。モデルはPythonのクラスでDBの表を表現したもので、Djangoがこれを読んでDBのテーブルを自動生成します。

この章のゴール: 6つのモデルを定義し、DBを作成し、管理画面でデータを登録できる状態にする。

---

## 2-1. カスタムユーザーを最初に決める（重要）

Djangoには標準のユーザー（`User`）がありますが、今回は「表示名」「ロール（クルー/リーダー）」を足したいので、**自前のユーザーモデル**に置き換えます。

> ⚠️ **これは必ず最初の `migrate` の前にやる**のが鉄則です。後から差し替えると大掛かりな作り直しになります。1章であえて `migrate` しなかったのはこのためです。

ファイル: `accounts/models.py`

```python
from django.contrib.auth.models import AbstractUser
from django.db import models


class User(AbstractUser):
    """アプリ利用者。クルー／リーダー共通。

    - ``username`` をログインID（リーダーが配布）として使う。
    - ``email`` は AbstractUser のものを任意項目のまま使う。
    - 退職者などは ``is_active``（AbstractUser標準）で無効化する。
    """

    class Role(models.TextChoices):
        STAFF = "staff", "クルー"
        LEADER = "leader", "リーダー"

    name = models.CharField("表示名", max_length=100)
    role = models.CharField(
        "ロール", max_length=16, choices=Role.choices, default=Role.STAFF
    )

    class Meta:
        verbose_name = "ユーザー"
        verbose_name_plural = "ユーザー"

    def __str__(self):
        return f"{self.name}（{self.username}）"

    @property
    def is_leader(self) -> bool:
        return self.role == self.Role.LEADER
```

**解説**

- `AbstractUser` を継承すると、`username` / `password` / `email` / `is_active` などの認証用フィールドをそのまま引き継げます。必要な `name` と `role` だけ足しています。
- `TextChoices` は「選択肢付きの文字列」を定義する便利機能。`role` には `"staff"` か `"leader"` のどちらかが入り、画面では「クルー」「リーダー」と表示されます。
- `is_leader` は「リーダーかどうか」をコード中で `user.is_leader` と書けるようにする**プロパティ**。後の章のアクセス制御で多用します。

### 設定でユーザーモデルを差し替える

ファイル: `shift_manager/settings.py`（`INSTALLED_APPS` の下あたりに追記）

```python
# 標準ユーザーの代わりに自前のUserを使う（最初のmigrate前に必須）
AUTH_USER_MODEL = 'accounts.User'
```

## 2-2. シフト関連のモデル

ファイル: `shifts/models.py`

少し長いですが、0章のER図をそのままコードにしたものです。全文を書き写してから解説を読んでください。

```python
from django.conf import settings
from django.core.exceptions import ValidationError
from django.db import models


def generate_time_choices():
    """8:00〜24:00を30分刻みで選択肢にする（"HH:MM" の文字列）。"""
    choices = []
    for hour in range(8, 24):
        choices.append((f"{hour:02d}:00", f"{hour:02d}:00"))
        choices.append((f"{hour:02d}:30", f"{hour:02d}:30"))
    choices.append(("24:00", "24:00"))
    return choices


TIME_CHOICES = generate_time_choices()


class ShiftPeriod(models.Model):
    """対象期間。リーダーが作る募集の単位。"""

    class Status(models.TextChoices):
        OPEN = "open", "募集中"
        CLOSED = "closed", "締切後"

    title = models.CharField("タイトル", max_length=100, blank=True)
    start_date = models.DateField("開始日")
    end_date = models.DateField("終了日")
    deadline = models.DateTimeField("締切日時")
    status = models.CharField(
        "状態", max_length=16, choices=Status.choices, default=Status.OPEN
    )
    created_by = models.ForeignKey(
        settings.AUTH_USER_MODEL,
        verbose_name="作成者",
        on_delete=models.PROTECT,
        related_name="created_periods",
    )
    created_at = models.DateTimeField("作成日時", auto_now_add=True)
    updated_at = models.DateTimeField("更新日時", auto_now=True)

    class Meta:
        verbose_name = "対象期間"
        verbose_name_plural = "対象期間"
        ordering = ["-start_date"]

    def __str__(self):
        return self.title or f"{self.start_date}〜{self.end_date}"

    def clean(self):
        if self.start_date and self.end_date and self.start_date > self.end_date:
            raise ValidationError({"end_date": "終了日は開始日以降にしてください。"})


class ShiftRequest(models.Model):
    """シフト希望（提出ヘッダ）。(period, user) で1件。行が無い＝未提出。"""

    period = models.ForeignKey(
        ShiftPeriod, verbose_name="対象期間",
        on_delete=models.CASCADE, related_name="requests",
    )
    user = models.ForeignKey(
        settings.AUTH_USER_MODEL, verbose_name="提出者",
        on_delete=models.CASCADE, related_name="shift_requests",
    )
    note = models.TextField("備考・連絡事項", blank=True)
    submitted_at = models.DateTimeField("最終提出日時", auto_now=True)
    created_at = models.DateTimeField("初回提出日時", auto_now_add=True)

    class Meta:
        verbose_name = "シフト希望"
        verbose_name_plural = "シフト希望"
        constraints = [
            models.UniqueConstraint(
                fields=["period", "user"], name="unique_request_per_user_period"
            )
        ]

    def __str__(self):
        return f"{self.user} / {self.period}"


class ShiftRequestDay(models.Model):
    """各日の希望（1日1枠）。"""

    request = models.ForeignKey(
        ShiftRequest, verbose_name="提出",
        on_delete=models.CASCADE, related_name="days",
    )
    work_date = models.DateField("対象日")
    is_available = models.BooleanField("出勤可", default=True)
    start_time = models.CharField(
        "開始時刻", max_length=5, null=True, blank=True, choices=TIME_CHOICES
    )
    end_time = models.CharField(
        "終了時刻", max_length=5, null=True, blank=True, choices=TIME_CHOICES
    )

    class Meta:
        verbose_name = "各日の希望"
        verbose_name_plural = "各日の希望"
        ordering = ["work_date"]
        constraints = [
            models.UniqueConstraint(
                fields=["request", "work_date"], name="unique_day_per_request"
            )
        ]

    def __str__(self):
        if not self.is_available:
            return f"{self.work_date}: 休み"
        return f"{self.work_date}: {self.start_time}〜{self.end_time}"


class ShiftRequestHistory(models.Model):
    """変更履歴。締切前の変更ごとにスナップショットを残す。"""

    request = models.ForeignKey(
        ShiftRequest, verbose_name="提出",
        on_delete=models.CASCADE, related_name="histories",
    )
    version_no = models.PositiveIntegerField("バージョン")
    snapshot = models.JSONField("内容スナップショット")
    created_at = models.DateTimeField("変更日時", auto_now_add=True)

    class Meta:
        verbose_name = "変更履歴"
        verbose_name_plural = "変更履歴"
        ordering = ["-version_no"]
        constraints = [
            models.UniqueConstraint(
                fields=["request", "version_no"], name="unique_version_per_request"
            )
        ]

    def __str__(self):
        return f"{self.request} v{self.version_no}"


def confirmed_shift_upload_path(instance, filename):
    return f"confirmed_shifts/period_{instance.period_id}/{filename}"


class ConfirmedShift(models.Model):
    """確定シフトPDF。リーダーがアップロードし、全員が閲覧する。"""

    period = models.ForeignKey(
        ShiftPeriod, verbose_name="対象期間",
        on_delete=models.CASCADE, related_name="confirmed_shifts",
    )
    file = models.FileField("ファイル", upload_to=confirmed_shift_upload_path)
    original_name = models.CharField("元ファイル名", max_length=255, blank=True)
    uploaded_by = models.ForeignKey(
        settings.AUTH_USER_MODEL, verbose_name="アップロード者",
        on_delete=models.PROTECT, related_name="uploaded_shifts",
    )
    uploaded_at = models.DateTimeField("アップロード日時", auto_now_add=True)

    class Meta:
        verbose_name = "確定シフト"
        verbose_name_plural = "確定シフト"
        ordering = ["-uploaded_at"]

    def __str__(self):
        return f"{self.period} 確定シフト"
```

**重要な書き方の解説**

- **`ForeignKey`（外部キー）** = 「1対多」の“多”側に置く参照。`ShiftRequestDay.request` は「この日データはどの提出に属するか」を指します。`related_name="days"` を付けると、逆向きに `request.days.all()` で日データ一覧を取れます。
- **`on_delete`** = 参照先が消えたときの挙動。
  - `CASCADE`: 一緒に消す（提出が消えたら各日データも消す）。
  - `PROTECT`: 消させない（期間を作った人を誤って消せないように守る）。
- **`settings.AUTH_USER_MODEL`** = ユーザーを参照するときの定石。`User` を直接importせずこう書くと、差し替えに強くなります。
- **`UniqueConstraint`** = 重複禁止。`(period, user)` を一意にして「同じ人が同じ期間に2件提出」を防ぎます。
- **`clean()`** = 入力チェック。`ShiftPeriod` では「終了日 < 開始日」を弾きます。
- **時刻は文字列 "HH:MM"**: `TimeField` ではなく `CharField` にしているのは **"24:00"（深夜0時）を扱いたい**から（Pythonの時刻型は24:00を表現できません）。`"08:00" < "17:00"` のように文字列でも前後比較が正しく効くのがポイント（常に2桁なので辞書順＝時刻順）。

## 2-3. 管理画面に登録する

Djangoには標準の管理画面（`/admin`）があり、登録すればGUIでデータを足せます。まずはこれでDB操作を体験します。

ファイル: `accounts/admin.py`

```python
from django.contrib import admin
from django.contrib.auth.admin import UserAdmin as BaseUserAdmin

from .models import User


@admin.register(User)
class UserAdmin(BaseUserAdmin):
    """カスタムUser用の管理画面。パスワードはハッシュ化して保存される。"""

    list_display = ("username", "name", "role", "is_active", "is_staff")
    list_filter = ("role", "is_active", "is_staff")
    search_fields = ("username", "name", "email")

    fieldsets = BaseUserAdmin.fieldsets + (
        ("アプリ情報", {"fields": ("name", "role")}),
    )
    add_fieldsets = BaseUserAdmin.add_fieldsets + (
        ("アプリ情報", {"fields": ("name", "role", "email")}),
    )
```

ファイル: `shifts/admin.py`

```python
from django.contrib import admin

from .models import (
    ConfirmedShift, ShiftPeriod, ShiftRequest,
    ShiftRequestDay, ShiftRequestHistory,
)


@admin.register(ShiftPeriod)
class ShiftPeriodAdmin(admin.ModelAdmin):
    list_display = ("__str__", "start_date", "end_date", "deadline", "status", "created_by")
    list_filter = ("status",)


class ShiftRequestDayInline(admin.TabularInline):
    model = ShiftRequestDay
    extra = 0


@admin.register(ShiftRequest)
class ShiftRequestAdmin(admin.ModelAdmin):
    list_display = ("user", "period", "submitted_at")
    list_filter = ("period",)
    inlines = [ShiftRequestDayInline]


@admin.register(ShiftRequestHistory)
class ShiftRequestHistoryAdmin(admin.ModelAdmin):
    list_display = ("request", "version_no", "created_at")


@admin.register(ConfirmedShift)
class ConfirmedShiftAdmin(admin.ModelAdmin):
    list_display = ("period", "original_name", "uploaded_by", "uploaded_at")
```

> 🔰 `@admin.register(...)` を付けるだけで、そのモデルが `/admin` に現れます。`inlines` は「親の編集画面に子を一緒に表示」する仕組みで、提出の中の各日データをまとめて見られます。

## 2-4. マイグレーション（DBを作る）

モデルの定義から、DBの設計図（マイグレーションファイル）を作り、それをDBに適用します。

```
$ python manage.py makemigrations
$ python manage.py migrate
```

- `makemigrations` … モデルの変更を検知して `accounts/migrations/0001_initial.py` などを生成。
- `migrate` … その設計図に従って実際にテーブルを作成（初回はDjango標準の認証・管理テーブルも作られます）。

> 🔰 今後モデルを変えるたびに、この2コマンドを実行します。「モデルを直したら makemigrations → migrate」とセットで覚えましょう。

### マイグレーションとは何か（しくみと考え方）

「マイグレーション」は **`models.py`（Pythonのモデル定義）と、実際のDBのテーブル構造を一致させるしくみ**です。役割の違いがポイントです。

| コマンド | すること | どこで実行 | Gitとの関係 |
| --- | --- | --- | --- |
| `makemigrations` | モデルの変更から**設計図ファイル**を生成（`shifts/migrations/0002_*.py` など） | **開発環境だけ** | 生成ファイルを**コミットする** |
| `migrate` | 設計図を**実際のDBに適用**（初回は全テーブル、2回目以降は**差分だけ**） | 開発・本番の**両方** | DBの中身は変える。Git管理外 |

```
[開発] models.py を変更 → makemigrations → 生成された 0002_*.py を git push
[本番] git pull（0002_*.py を受け取る） → migrate（差分だけDBに適用）
```

**一番大事な性質：`migrate` は既存データを保ったまま、構造の差分だけを当てる**。例えば「シフトに `memo` 列を追加」しても、既存の行は消えず空の `memo` 列が足されるだけです。だから本番更新で**データは消えません**（[13章](13_本番運用.md)・[19章](19_Docker化.md) の更新手順）。

> 🔰 マイグレーションファイルは「DBの変更履歴」。`0001`→`0002`→… と積み重なり、`migrate` は**まだ当てていない分**だけを順に適用します。`python manage.py showmigrations` で適用済み（[X]）と未適用（[ ]）を一覧できます。
>
> ⚠️ **`makemigrations` は本番では実行しない**のが原則。設計図は開発で作って**コミット**し、本番は `migrate` で**適用するだけ**。本番で `makemigrations` すると環境ごとに設計図がズレて事故のもとです。

## 2-5. 動作確認

### ✅ 管理画面でデータを登録

```
$ python manage.py runserver
```

<http://127.0.0.1:8000/admin/> を開き、先ほどの管理者でログイン。

1. 「ユーザー」を開き、自分のアカウントを編集して **表示名** を入れ、**ロール=リーダー** に変更して保存。
2. 「対象期間」を1件追加（タイトル・開始/終了日・締切日時を入力）。
3. 「シフト希望」を1件追加し、インラインで「各日の希望」を入れてみる。

エラーなく保存できれば、モデルは正しく作れています。

## 2-6. つまずきポイント

- **`makemigrations` で「AUTH_USER_MODEL refers to model that has not been installed」** → `accounts` を `INSTALLED_APPS` に入れ忘れ、または `AUTH_USER_MODEL` のスペルミス。
- **すでに `migrate` してしまった後でカスタムユーザーに変えたい** → 学習用なら `db.sqlite3` と各アプリの `migrations/0001_*.py`（`__init__.py`は残す）を削除して作り直すのが早いです。⚠️ これは **DBごと消して作り直す開発専用の荒技**。**本番では絶対にやらない**（全データが消えます）。本番でモデルを直すときは「`makemigrations`（開発）→ コミット → 本番で `migrate`」の正規ルートのみ。
- **`createsuperuser` で `name` を聞かれない** → 標準では聞かれません。作成後に /admin で表示名を設定すればOK。
