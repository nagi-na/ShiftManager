# 11章 自動テスト ― 壊れていないことを毎回確かめる

ここまでで一通り動くアプリができました。これから機能を足したり直したりするたびに、「**前に動いていた部分を壊していないか**」を確認したくなります。毎回ブラウザで全画面をぽちぽち確認するのは大変で、見落としも起きます。

そこで **自動テスト** を書きます。一度書けば `python manage.py test` の一発で、権限・提出・締切のルールが今も正しく動くかを数十秒で確認できます。

> 🔰 この章は他の章と少し性質が違い、アプリの機能は増えません。代わりに「これまで作った機能を守る安全網」を用意します。実務では機能と同じくらい重要なスキルです。

---

## 11-0. 何をテストするか ― 優先順位の考え方

すべてを100%テストする必要はありません。**「壊れたら困る・目視で確認しづらいロジック」から**書くのがコツです。このアプリで最優先なのは次の3つです。

| 観点 | 対象 | なぜ重要か |
| --- | --- | --- |
| ① 権限 | `can_manage` 等・`manager_required`/`admin_required` | 誰がどの画面に入れるか。間違えると情報漏れ・事故 |
| ② 提出フロー | `shift_submit` ビュー | 提出・休み・時刻チェック・変更履歴の中心機能 |
| ③ 締切ポリシー | `_submission_permission` 等 | 条件分岐が多く、目視で全パターン確認しづらい |

逆に、テンプレートの見た目や Bootstrap の表示はテストしません（コスパが悪い）。

---

## 11-1. Django のテストの仕組み

Django はテスト用の道具を標準で持っています。テストは各アプリの `tests.py` に書き、`python manage.py test` で実行します。

押さえるべき部品は3つです。

- **`TestCase`**: テストをまとめるクラス。`test_` で始まるメソッドが1件のテストになる。
- **テスト専用DB**: テスト実行時に**本物とは別の空のDB**が自動で作られ、終わると破棄される。さらに各テストメソッドごとに変更が**自動でロールバック**されるので、テスト同士が干渉しない。本番データを汚す心配はない。
- **テストクライアント `self.client`**: ブラウザの代わりにリクエストを送る。`self.client.get(url)` / `self.client.post(url, data)` でビューを呼び、返ってきたレスポンスを検査できる。

検査には `assert` 系メソッドを使います。

| メソッド | 意味 |
| --- | --- |
| `assertTrue(x)` / `assertFalse(x)` | x が真／偽か |
| `assertEqual(a, b)` | a == b か |
| `assertRedirects(resp, url)` | resp が url へリダイレクトしたか |
| `self.client.force_login(user)` | パスワード入力を省いてログイン状態にする |
| `obj.refresh_from_db()` | DBから最新状態を読み直す（更新が反映されたか確認するため） |

## 11-2. 権限のテスト（accounts）

まず、ロールごとのアクセス制御を確認します。共通のユーザー生成を**ヘルパー関数**にまとめておくと、各テストがすっきりします。

ファイル: `accounts/tests.py`

```python
from django.test import TestCase
from django.urls import reverse

from .models import User


def make_user(username, role, password="pass12345", **extra):
    """指定ロールのユーザーを作るテスト用ヘルパー。"""
    user = User.objects.create_user(username=username, password=password, **extra)
    user.role = role
    user.save(update_fields=["role"])
    return user
```

### 権限プロパティの単体テスト

まずDBや画面に依存しない、モデルのプロパティだけを確かめます。一番小さく速いテストです。

```python
class RolePropertyTests(TestCase):
    def test_admin_has_all_permissions(self):
        admin = make_user("admin", User.Role.ADMIN)
        self.assertTrue(admin.is_admin)
        self.assertTrue(admin.can_manage)
        self.assertTrue(admin.can_manage_accounts)

    def test_leader_can_manage_but_not_accounts(self):
        leader = make_user("leader", User.Role.LEADER)
        self.assertTrue(leader.can_manage)
        self.assertFalse(leader.can_manage_accounts)  # ここがリーダーの境界

    def test_crew_cannot_manage(self):
        crew = make_user("crew", User.Role.CREW)
        self.assertFalse(crew.can_manage)
        self.assertFalse(crew.can_manage_accounts)
```

### 画面のアクセス制御テスト

次に、実際に画面（ビュー）へアクセスして、ロール別に許可／拒否されるかを確認します。`reverse("名前")` でURL名から実URLを取り出すと、URLが変わってもテストが壊れません。

```python
class ManagerAccessTests(TestCase):
    def setUp(self):
        self.url = reverse("manage_periods")
        self.admin = make_user("admin", User.Role.ADMIN)
        self.leader = make_user("leader", User.Role.LEADER)
        self.crew = make_user("crew", User.Role.CREW)

    def test_anonymous_redirected_to_login(self):
        resp = self.client.get(self.url)
        self.assertRedirects(resp, reverse("login"))   # 未ログインはログインへ

    def test_crew_denied_redirected_home(self):
        self.client.force_login(self.crew)
        resp = self.client.get(self.url)
        self.assertRedirects(resp, reverse("home"))     # 権限不足はホームへ

    def test_leader_allowed(self):
        self.client.force_login(self.leader)
        self.assertEqual(self.client.get(self.url).status_code, 200)
```

> 🔰 `setUp` は各テストメソッドの**前に毎回**呼ばれる準備処理です。共通のユーザー作成などをここに置きます。

アカウント管理画面は管理者専用なので、**リーダーでも弾かれる**点が `manager_required` との違いです。ここを別クラスで明示的にテストします（`AdminAccessTests`）。`test_leader_denied` で `assertRedirects(resp, reverse("home"))` を確認します。

### ロックアウト防止のテスト（最重要）

「自分自身を無効化・降格・削除できない」ガードは、間違えると**管理者が締め出される**重大バグになります。ここは特に丁寧にテストします。

```python
class LockoutPreventionTests(TestCase):
    def setUp(self):
        self.admin = make_user("admin", User.Role.ADMIN, name="管理者")
        self.crew = make_user("crew", User.Role.CREW, name="クルー")
        self.client.force_login(self.admin)

    def test_cannot_deactivate_self(self):
        self.client.post(reverse("manage_account_toggle", args=[self.admin.pk]))
        self.admin.refresh_from_db()
        self.assertTrue(self.admin.is_active)   # 無効化されず有効のまま

    def test_can_deactivate_other_user(self):
        # 対照テスト: 他人は無効化できる ＝ 制限は「自分」だけに効いている
        self.client.post(reverse("manage_account_toggle", args=[self.crew.pk]))
        self.crew.refresh_from_db()
        self.assertFalse(self.crew.is_active)
```

> 🔰 **対照テスト**を必ず添えましょう。「禁止操作が止まる」だけだと、ガードが効きすぎて正常操作まで止めていても気づけません。「正常操作は通る」も確認してはじめて安心できます。

## 11-3. 締切ポリシーのテスト（純粋関数を直接呼ぶ）

締切後の扱いは3ポリシー × 提出済み/未提出 × 締切前後 …と分岐が多く、ビュー経由で全部試すと大変です。こういう**入力→出力がはっきりした関数**は、ビューを通さず**関数を直接呼ぶ**のが最短です。

`shifts/views.py` の `_submission_permission(period, now, submitted)` は `(新規提出できるか, 編集できるか)` を返します。これをそのまま検査します。

ファイル: `shifts/tests.py`

```python
from datetime import timedelta
from django.test import TestCase
from django.utils import timezone

from accounts.models import User
from .models import ShiftPeriod
from .views import _submission_permission, _is_closed

Policy = ShiftPeriod.PostDeadlinePolicy
Status = ShiftPeriod.Status


def make_period(creator, deadline, *, status=Status.OPEN,
                policy=Policy.LATE_SUBMIT, edit_deadline=None):
    today = timezone.localdate()
    return ShiftPeriod.objects.create(
        title="テスト期間", start_date=today, end_date=today,
        deadline=deadline, status=status,
        post_deadline_policy=policy, edit_deadline=edit_deadline,
        created_by=creator,
    )


class DeadlineLogicTests(TestCase):
    def setUp(self):
        self.creator = User.objects.create_user("leader", password="x")
        self.now = timezone.now()
        self.past = self.now - timedelta(hours=1)    # 締切=過去 → 締切後
        self.future = self.now + timedelta(hours=1)  # 締切=未来 → 締切前

    def test_before_deadline_anyone_can_submit_and_edit(self):
        period = make_period(self.creator, self.future)
        self.assertEqual(_submission_permission(period, self.now, False), (True, True))

    def test_after_deadline_late_submit(self):
        # 未提出者だけ提出可・編集は不可
        period = make_period(self.creator, self.past, policy=Policy.LATE_SUBMIT)
        self.assertEqual(_submission_permission(period, self.now, False), (True, False))
        self.assertEqual(_submission_permission(period, self.now, True), (False, False))

    def test_after_deadline_lock_all(self):
        period = make_period(self.creator, self.past, policy=Policy.LOCK_ALL)
        self.assertEqual(_submission_permission(period, self.now, False), (False, False))
```

> 🔰 締切の「前／後」は、`deadline` を `now ± 1時間` に置くだけで作れます。実際の現在時刻に依存しないよう、`now` を `setUp` で固定して関数に渡しているのがポイントです。

`EDITABLE`（締切後も編集可）は「編集最終期限なし＝締め切るまで可」「期限内＝可」「期限切れ＝不可」の3分岐があるので、それぞれ1ケースずつ書きます。実ファイルの `DeadlineLogicTests` に全パターンが入っています。

## 11-4. 提出フローのテスト（フォームセットのPOST）

提出画面 `shift_submit` は **formset**（各日の希望フォームの集まり）を受け取ります。テストから formset を POST するには、`day-TOTAL_FORMS` などの**管理用フィールド**を含める必要があります。毎回書くと長いので、データ組み立てを**ヘルパー関数**にします。

```python
def formset_post_data(days, note=""):
    """days: [{"work_date": date, "day_off": bool, "start": "HH:MM", "end": "HH:MM"}]"""
    data = {
        "day-TOTAL_FORMS": str(len(days)),
        "day-INITIAL_FORMS": "0",
        "day-MIN_NUM_FORMS": "0",
        "day-MAX_NUM_FORMS": "1000",
        "note": note,
    }
    for i, d in enumerate(days):
        data[f"day-{i}-work_date"] = d["work_date"].isoformat()
        if d.get("day_off"):
            data[f"day-{i}-is_day_off"] = "on"   # チェックボックスはオンのとき "on"
        if d.get("start"):
            data[f"day-{i}-start_time"] = d["start"]
        if d.get("end"):
            data[f"day-{i}-end_time"] = d["end"]
    return data
```

これを使って、提出の正常系・異常系をテストします。

```python
class SubmitFlowTests(TestCase):
    def setUp(self):
        self.leader = User.objects.create_user("leader", password="x")
        self.crew = User.objects.create_user("crew", password="x")
        self.crew.role = User.Role.CREW
        self.crew.save(update_fields=["role"])
        today = timezone.localdate()
        self.period = make_period_2days(self.leader, today)   # 締切は未来＝募集中
        self.d0, self.d1 = today, today + timedelta(days=1)
        self.client.force_login(self.crew)
        self.url = reverse("shift_submit", args=[self.period.pk])

    def test_submit_creates_request_days_and_history(self):
        data = formset_post_data([
            {"work_date": self.d0, "start": "09:00", "end": "17:00"},
            {"work_date": self.d1, "start": "10:00", "end": "18:00"},
        ], note="よろしくお願いします")
        resp = self.client.post(self.url, data)
        self.assertRedirects(resp, reverse("home"))

        req = ShiftRequest.objects.get(period=self.period, user=self.crew)
        self.assertEqual(req.days.count(), 2)
        self.assertEqual(req.histories.count(), 1)   # 変更履歴 v1 が残る

    def test_working_day_without_times_is_rejected(self):
        # 出勤日なのに時刻なし → フォーム不正 → 保存されない
        data = formset_post_data([
            {"work_date": self.d0},  # 休みでないのに時刻未選択
            {"work_date": self.d1, "start": "10:00", "end": "18:00"},
        ])
        resp = self.client.post(self.url, data)
        self.assertEqual(resp.status_code, 200)  # リダイレクトせず再表示
        self.assertFalse(
            ShiftRequest.objects.filter(period=self.period, user=self.crew).exists()
        )
```

実ファイルでは他に、**「休み」は時刻なしで保存される**／**開始>終了は拒否**／**再提出で履歴がv2に増える**／**締切後LOCK_ALLはPOSTしても保存されない**／**非表示期間はクルーがアクセス不可**、もテストしています。

> ⚠️ formset の管理用フィールド（`TOTAL_FORMS` 等）を忘れると `ManagementForm data is missing` というエラーになります。テストでフォームセットを送るときの定番のつまずきです。

## 11-5. 実行と結果の読み方

```
$ ./venv/bin/python manage.py test          # 全テスト
$ ./venv/bin/python manage.py test accounts # accounts だけ
$ ./venv/bin/python manage.py test shifts -v2  # 詳細表示（テスト名が1件ずつ出る）
```

最後に次のように出れば全て成功です。

```
Ran 31 tests in 28.5s

OK
```

失敗すると `FAIL`（assert が不一致）や `ERROR`（例外）と、どのテストのどの行かが表示されます。**まずその1件を読む**のが解決の早道です。

> 🔰 開発の習慣にしましょう。「コードを直したら `manage.py test`」を癖にすると、壊した瞬間に気づけます。これが自動テストの一番の価値です。

## つまずきポイント

- **`ManagementForm data is missing`**: formset を POST するとき `day-TOTAL_FORMS` 等を入れ忘れている（11-4のヘルパー参照）。
- **`refresh_from_db()` を忘れる**: POST 後にオブジェクトを検査しても、メモリ上の古い値のままで変化が見えない。DB更新を確認するときは必ず読み直す。
- **テストが本番DBを使ってしまう不安**: 心配無用。テストは専用DBを自動生成し、終了後に破棄する。`db.sqlite3` は触られない。
- **`reverse("名前")` でエラー**: `urls.py` の `name=` と綴りが一致しているか確認。URL名でリンクすると、パスを変えてもテストとテンプレートが壊れない。
