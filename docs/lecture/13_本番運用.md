# 13章 本番運用

完成したアプリを、開発用の `runserver` から **本番そっくりの構成**へ載せ替えます。この章は、実際に手を動かして次の構成を組み上げる実践記録です。

```
ブラウザ → Nginx(:80) ┬─ /static/            → ファイルを直接配信
                       ├─ /protected/(internal) → 認証OK時だけ確定シフトを送信(7-7)
                       └─ それ以外            → Gunicorn(:8000) → Django
```

> 🔰 この章は **まず自分のPC（WSL/Linux）で練習**する前提で書いています。実サーバーがなくても本番の流れを安全に体験でき、失敗してもPC内で完結します。DBは開発と同じ **SQLite のまま**進めます（同時アクセスが多くなったら MySQL 化＝14章）。
>
> ⚠️ ただし [12章](12_開発編のまとめ.md) のとおり、本格運用では **開発環境と本番環境は分ける**のが原則です（理想は本番を別ホストへ）。同じPCで練習する場合も、**開発のデータ・設定を本番と混ぜない**よう意識してください。

### 前提環境

本番編は次の環境で動作確認しています。コマンド（特に `apt` と `systemd`）はこれに依存します。

- **OS**: Ubuntu 24.04 / WSL2（Debian系。`apt` が使える）
- **サービス管理**: `systemd` が有効（`ps -p 1 -o comm=` が `systemd` を返す）
- macOS / Windowsネイティブの方は、**WSL2 もしくは Linux の仮想マシン**を用意して進めるのが確実です。

### 置換早見表（重要）

本書のコマンドは、練習環境の**実際の値をそのまま**載せています。あなたの環境では下表のように読み替えてください（特に **パスとユーザー名**）。

| 本書の値 | 意味 | あなたの値（例） |
| --- | --- | --- |
| `/home/nagin/ShiftManager` | アプリの絶対パス | `/home/<あなた>/ShiftManager`（`pwd` で確認） |
| `nagin` | 実行ユーザー名 | `whoami` の結果 |
| `localhost` / `127.0.0.1` | 公開ホスト名 | 練習はそのまま。公開時は `shift.example.com` 等 |
| `shiftmanager`（サービス名） | systemdサービス名 | そのままでOK（任意に変更可） |

> 🔰 以降、`/home/nagin/ShiftManager` や `User=nagin` が出てきたら、上表に従って自分の値へ置き換えてください。`deploy/` 内の設定ファイルも同様です。

## 13-1. 開発と本番の違いを押さえる

| 項目 | 開発（これまで） | 本番 |
| --- | --- | --- |
| 起動 | `runserver` | **Gunicorn**（WSGIサーバー） |
| 前段 | なし | **Nginx**（リバースプロキシ／静的配信） |
| `DEBUG` | `True` | **`False`** |
| `SECRET_KEY` | 自動生成 | **環境変数で秘匿** |
| `ALLOWED_HOSTS` | 空でOK | **公開するホスト名/IP** |
| 静的ファイル | Djangoが自動配信 | `collectstatic` → **Nginxが配信** |

要点は「**`DEBUG=False` にする**」「**秘密情報をコードから出す**」「**静的/メディアはNginxに任せる**」の3つです。

## 13-2. 設定を環境変数で切り替える

`settings.py` を本番用と開発用に分けず、**環境変数で値を差し替える**方式にします。普段の開発はこれまで通り、環境変数を渡したときだけ本番モードになります。

ファイル: `shift_manager/settings.py`（該当箇所）

```python
import os
from pathlib import Path

from django.core.management.utils import get_random_secret_key

BASE_DIR = Path(__file__).resolve().parent.parent

# 本番では環境変数 DJANGO_SECRET_KEY を必ず設定する。
# 未設定なら起動ごとにランダム生成（開発用。再起動でセッションは失効する）。
SECRET_KEY = os.environ.get('DJANGO_SECRET_KEY', get_random_secret_key())

# 環境変数 DJANGO_DEBUG が "0" のとき本番モード（既定は開発モード=True）。
DEBUG = os.environ.get('DJANGO_DEBUG', '1') != '0'

# 本番（DEBUG=False）では、接続を許可するホスト名をカンマ区切りで指定する。
# 例: DJANGO_ALLOWED_HOSTS="example.com,127.0.0.1"
ALLOWED_HOSTS = (
    [h.strip() for h in os.environ.get('DJANGO_ALLOWED_HOSTS', '').split(',') if h.strip()]
    if not DEBUG
    else []
)
```

**解説**

- **`SECRET_KEY`**: セッションやCSRFの署名に使う鍵。漏れると改ざんされうるので、コード/Gitには載せず環境変数で渡します。開発では `get_random_secret_key()` が毎回作る使い捨ての鍵で十分です（[10章](10_GitHubで管理.md) の秘密情報の扱いも参照）。
- **`DEBUG=False` が本番の絶対条件**: `True` のままだとエラー画面に**ソースや設定が丸見え**になります。さらに `DEBUG=False` では **Django は静的ファイルを配信しなくなります**（だから Nginx が要る、という後の伏線）。
- **`ALLOWED_HOSTS`**: `DEBUG=False` のとき、ここに無いホスト名でのアクセスは 400 で拒否されます（なりすまし対策）。

開発モードのまま壊れていないか確認します。

```
$ ./venv/bin/python manage.py check
System check identified no issues (0 silenced).
```

## 13-3. 静的ファイルを集める（collectstatic）

本番では Nginx が CSS/JS を配信します。そのために、各アプリや Django 管理画面に散らばる静的ファイルを **`STATIC_ROOT`（`staticfiles/`）の1か所へ集約**します。`settings.py` には設定済みです。

```python
STATIC_URL = 'static/'
STATICFILES_DIRS = [BASE_DIR / 'static']
STATIC_ROOT = BASE_DIR / 'staticfiles'   # collectstatic の集約先（本番用）
```

集めます（本番モードで実行）。

```
$ DJANGO_DEBUG=0 DJANGO_ALLOWED_HOSTS=127.0.0.1,localhost \
    ./venv/bin/python manage.py collectstatic --noinput
127 static files copied to '.../staticfiles'.
```

> 🔰 本アプリの見た目は CDN の Bootstrap なので自前CSSはほぼありませんが、**Django管理画面(/admin)のCSS** などのために `collectstatic` は本番で必須です。`staticfiles/` は自動生成物なので `.gitignore` 済みです。

## 13-4. Gunicorn で動かす

`runserver` は開発専用です。本番は **Gunicorn**（WSGIアプリケーションサーバー）で動かします。

```
$ ./venv/bin/pip install gunicorn
```

`requirements.txt` にも追記しておきます。

```
gunicorn>=23,<27
```

本番モードで起動してみます。

```
$ DJANGO_DEBUG=0 DJANGO_ALLOWED_HOSTS=127.0.0.1,localhost \
    ./venv/bin/gunicorn shift_manager.wsgi:application --bind 127.0.0.1:8000
```

- `shift_manager.wsgi:application` … プロジェクトに最初からある `wsgi.py` の `application`。
- `127.0.0.1:8000` … ローカルだけで待ち受け、外からは直接触らせず Nginx 経由にします。

別ターミナルから確認します。

```
$ curl -s -o /dev/null -w "%{http_code}\n" http://127.0.0.1:8000/login/
200
```

### ✅ ここで大事な確認 ― Gunicorn は静的ファイルを配信しない

```
$ curl -s -o /dev/null -w "%{http_code}\n" http://127.0.0.1:8000/static/admin/css/base.css
404
```

`DEBUG=False` の Gunicorn 単体では静的ファイルが **404**。これが「Nginx を前に置く理由」です。

## 13-5. Nginx を前段に置く

Nginx が外からの受け口になり、`/static/` は自分で配信し、それ以外を Gunicorn へ中継します。確定シフト（media）は **7-7 の認証付き配信**に合わせて、外から直接は叩けない `internal` な `/protected/` 経由にします。設定はリポジトリの **`deploy/nginx-shiftmanager.conf`** に用意してあります。

ファイル: `deploy/nginx-shiftmanager.conf`

```nginx
server {
    listen 80;
    server_name localhost 127.0.0.1;

    client_max_body_size 12m;   # 確定シフトPDF(最大10MB)に余裕を持たせる

    location /static/ {
        alias /home/nagin/ShiftManager/staticfiles/;
        access_log off;
        expires 7d;
    }
    # 確定シフト等のメディア。internal なので外部から直接は開けない。
    # Django(@login_required)が X-Accel-Redirect: /protected/... を返したときだけ送信。
    location /protected/ {
        internal;
        alias /home/nagin/ShiftManager/media/;
    }
    location / {
        proxy_pass http://127.0.0.1:8000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

インストールして有効化します（`sudo` が必要）。

```
$ sudo apt-get update && sudo apt-get install -y nginx
$ sudo cp deploy/nginx-shiftmanager.conf /etc/nginx/sites-available/shiftmanager
$ sudo ln -sf /etc/nginx/sites-available/shiftmanager /etc/nginx/sites-enabled/shiftmanager
$ sudo rm -f /etc/nginx/sites-enabled/default     # Nginxの初期ページを無効化
$ sudo nginx -t                                   # 設定の文法チェック
$ sudo systemctl reload nginx
```

確認します（Gunicorn は起動したまま、ポート80＝Nginx を叩く）。

```
$ curl -s -o /dev/null -w "%{http_code}\n" http://localhost/login/
200
```

**解説**

- `/static/` は Nginx が `alias` のパスから直接ファイルを返します（速い・Django を煩わせない）。
- `/protected/` は **`internal`** 指定なので、ブラウザから直接 `/protected/...` を開いても 404 になります。Django のビュー（`@login_required`）が `X-Accel-Redirect: /protected/...` を返したときだけ、Nginx が実ファイルを送信します（＝認証は Django・転送は Nginx／7-7）。`/media/` を直接公開していた頃は**誰でもDL可能**だったので、この形に直してあります。
- それ以外は `proxy_pass` で Gunicorn へ。`X-Forwarded-*` ヘッダで「本当のアクセス元・スキーム」を Django に伝えます。
- `client_max_body_size` … アップロード上限。これが小さいと PDF アップロードが Nginx 側で弾かれます。

### ⚠️ つまずき実例: 静的ファイルが 403 になる

```
$ curl -s -o /dev/null -w "%{http_code}\n" http://localhost/static/admin/css/base.css
403   ← Forbidden
```

原因は **ファイルの権限**でした。Nginx は `www-data` というユーザーで動きます。ところがアプリが `/home/nagin/` 配下にあり、ホームディレクトリの権限が `750`（`drwxr-x---`）だと、`www-data` はそこを**通り抜けられず**静的ファイルに到達できません。

```
$ namei -l /home/nagin/ShiftManager/staticfiles/admin/css/base.css
  drwxr-x---  nagin nagin  /home/nagin    ← ここで www-data が弾かれる
  drwxr-xr-x  nagin nagin  ShiftManager   （以下は誰でも読める）
  ...
```

ホームに「通過（traverse）」だけを許可して解決します（中の一覧を許す `r` は付けない最小変更）。

```
$ sudo chmod o+x /home/nagin
$ curl -s -o /dev/null -w "%{http_code}\n" http://localhost/static/admin/css/base.css
200   ← 解決
```

> 🔰 実サーバーでは、アプリを `/home` 配下ではなく `/srv/shiftmanager` や `/var/www` など **Webサーバーが読める場所**に置くのが定石です。今回は練習なので最小修正で通しました。

## 13-6. Gunicorn を systemd で常駐させる

PC（サーバー）再起動後も自動で立ち上がり、落ちても再起動されるよう、サービス化します。定義はリポジトリの **`deploy/gunicorn-shiftmanager.service`** にあります。

ファイル: `deploy/gunicorn-shiftmanager.service`

```ini
[Unit]
Description=ShiftManager Gunicorn daemon
After=network.target

[Service]
User=nagin
Group=nagin
WorkingDirectory=/home/nagin/ShiftManager
Environment=DJANGO_DEBUG=0
Environment=DJANGO_ALLOWED_HOSTS=localhost,127.0.0.1
Environment=DJANGO_SECRET_KEY=ここに本番用の長いランダム文字列を貼る
ExecStart=/home/nagin/ShiftManager/venv/bin/gunicorn \
    shift_manager.wsgi:application \
    --bind 127.0.0.1:8000 \
    --workers 3
Restart=on-failure

[Install]
WantedBy=multi-user.target
```

手動で起動していた Gunicorn は止めてから（ポート8000の衝突を避ける）、サービスを登録します。**SECRET_KEY は固定値を生成して、root権限で保護される `/etc/systemd/system/` のファイルへ書き込み**ます（リポジトリには載せない）。

> ⚠️ **SECRET_KEY は「全ワーカー共通の固定値」を必ず設定する。** Gunicorn は複数ワーカー（`--workers 3`）で動くため、`DJANGO_SECRET_KEY` を渡し忘れると各ワーカーが起動時に **別々のランダム鍵** を作り、セッション署名が食い違って利用者が **勝手にログアウト** します（公開後に最も起きやすい不具合）。
>
> 🔰 **鍵は英数字だけにする。** systemd の `Environment=` は `%`（specifier）と `$`（変数展開）を特別扱いするため、`%`/`$` を含む鍵を書くとその行が無効化され、結局**鍵が渡らない**（＝上のログアウトが起きる）ことがあります。`get_random_secret_key()` は記号を含みうるので、ここでは**英数字のみ**で生成します。

```
$ pkill -f "gunicorn shift_manager.wsgi"     # 手動Gunicornがいれば停止

# 英数字だけの固定SECRET_KEYを生成し、テンプレートの該当箇所に埋め込んで書き込む（鍵は画面に出さない）
$ KEY=$(python3 -c "import secrets,string; print(''.join(secrets.choice(string.ascii_letters+string.digits) for _ in range(64)))")
$ sed "s#ここに本番用の長いランダム文字列を貼る#$KEY#" deploy/gunicorn-shiftmanager.service \
    | sudo tee /etc/systemd/system/shiftmanager.service >/dev/null
$ sudo chmod 640 /etc/systemd/system/shiftmanager.service   # 鍵を含むので保護

$ sudo systemctl daemon-reload
$ sudo systemctl enable --now shiftmanager    # 今すぐ起動＋次回も自動起動
$ sudo systemctl status shiftmanager --no-pager
● shiftmanager.service - ShiftManager Gunicorn daemon
     Loaded: loaded (...; enabled; ...)
     Active: active (running) ...
```

`Active: active (running)` かつ `enabled` になれば常駐化成功です。続いて **全ワーカーに鍵が渡っているか**（ログアウト予防の確認）と、Nginx 経由の応答を確認します。

```
# 各ワーカーで「1」が並べば、全員に同じ鍵が渡っている（バラバラ／0ならログアウトの原因）
$ for pid in $(pgrep -f "gunicorn shift_manager"); do
    tr '\0' '\n' < /proc/$pid/environ | grep -c '^DJANGO_SECRET_KEY='
  done
$ curl -s -o /dev/null -w "%{http_code}\n" http://localhost/login/                    # 200
$ curl -s -o /dev/null -w "%{http_code}\n" http://localhost/static/admin/css/base.css # 200
```

> 🔰 秘密情報（`SECRET_KEY` など）はこの**サービス定義（root権限・`chmod 640`）**に置きます。リポジトリ内の `deploy/gunicorn-shiftmanager.service` はプレースホルダのまま＝Gitに鍵を載せません。`%`/`$` を含む鍵をどうしても使いたい場合は `EnvironmentFile=`（値をそのまま読む）に書く方法もあります。

## 13-7. 公開前チェック（check --deploy）

Django には本番設定の安全性を点検するコマンドがあります。

```
$ DJANGO_DEBUG=0 DJANGO_ALLOWED_HOSTS=localhost,127.0.0.1 DJANGO_SECRET_KEY=dummy \
    ./venv/bin/python manage.py check --deploy
```

今回出た警告と読み解き:

| 警告 | 意味 | 対応 |
| --- | --- | --- |
| W009 SECRET_KEY | 鍵が短い/自動生成 | **本番用の長い鍵**を環境変数で渡せば解消（上のdummyは確認用） |
| W004 HSTS | HTTPS強制ヘッダ未設定 | HTTPS導入時に対応 |
| W008 SSL_REDIRECT | HTTP→HTTPS転送 | 同上 |
| W012/W016 Cookie Secure | Cookieを暗号化通信限定に | 同上 |

W004・W008・W012・W016 はすべて **HTTPS（暗号化通信）関連**で、HTTPのみの練習環境では想定どおりです。実サーバーで公開するときに対応します（13-9）。

## 13-8. コードを更新したときの手順

以降、アプリを直したらこの流れで反映します（`deploy/README.md` にも記載）。

```
$ git pull                                              # 別PCの更新を取り込む場合
$ ./venv/bin/python manage.py migrate                   # モデル変更があれば
$ DJANGO_DEBUG=0 ./venv/bin/python manage.py collectstatic --noinput  # 静的変更があれば
$ sudo systemctl restart shiftmanager                   # アプリを再起動
```

ログの見方:

```
$ sudo journalctl -u shiftmanager -f      # Gunicorn(アプリ)のログ
$ sudo tail -f /var/log/nginx/error.log   # Nginxのログ
```

> 🔰 **別環境でまったく同じ構成を再現したいとき**は、依存をバージョンごと固定したファイルを作っておくと確実です。`requirements.txt` は範囲指定（`>=`）なので、`./venv/bin/pip freeze > requirements.lock` で実際のバージョンを記録し、移行先で `pip install -r requirements.lock` を使います。

## 13-9. 実サーバーへ出すとき（このあと）

このPCでの練習と実公開の差分です。

- **パス・ホスト名**: `deploy/` 各ファイルの `/home/nagin/ShiftManager`・`User=`・`server_name`・`DJANGO_ALLOWED_HOSTS` を実際の値へ。アプリは `/srv` 等のWeb可読な場所へ置くと権限問題を避けられる。
- **HTTPS化（公開なら必須級）**: `certbot`（Let's Encrypt）で証明書を取り、443で配信、80→443へリダイレクト。併せて settings で `SECURE_SSL_REDIRECT` / `SESSION_COOKIE_SECURE` / `CSRF_COOKIE_SECURE = True`、`CSRF_TRUSTED_ORIGINS = ["https://あなたのドメイン"]` を設定（13-7の警告が消える）。
- **同時アクセスが増えたら MySQL 化**: `pip install mysqlclient`（OS側に `default-libmysqlclient-dev` 等が必要）し、`DATABASES` を環境変数で切り替え、新DBに `migrate` → `createsuperuser`。
- **学内サーバーの場合**: 外部公開の可否・ポート開放・ドメインは管理者の規約に従う。まずは学内ネットワーク限定が無難。
- **バックアップ**: SQLiteなら `db.sqlite3`、アップロードPDFは `media/` をまるごと定期取得。

## つまずきポイント

- **`DEBUG=False` にしたら 400 (Bad Request)** → `ALLOWED_HOSTS` にアクセス元のホスト名/IPが入っていない。
- **静的ファイルが 403** → ファイル/ディレクトリの権限。Nginx の `www-data` が到達できるか（13-5の `chmod o+x`、または配置場所の見直し）。
- **CSSが当たらない（管理画面が崩れる）** → `collectstatic` 未実行、または Nginx の `/static/` の `alias` パス違い。
- **systemd 起動失敗（Address already in use）** → 手動 Gunicorn が残ってポート8000を握っている。`pkill -f gunicorn` してから起動。
- **PDFがアップロードできない** → Nginx の `client_max_body_size` を確認。
- **POSTが403 (CSRF)** → HTTPS化したら `CSRF_TRUSTED_ORIGINS` を設定。

---

## この章のまとめ

- 設定を環境変数で切り替え（`DEBUG`・`ALLOWED_HOSTS`・`SECRET_KEY`）、`collectstatic` で静的ファイルを集約した。
- Gunicorn でアプリを動かし、Nginx を前段に置いて静的配信＋中継する構成を組んだ。
- 静的ファイルの 403（権限）を実際に踏んで解決し、systemd で常駐化（自動起動）した。
- `check --deploy` で残課題（HTTPS関連）を把握し、実サーバーへの差分を整理した。
