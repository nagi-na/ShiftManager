# 17章 名前付きトンネルでの恒久公開と安全な運用設定

> 🔰 この章も **運用編** です。[16章](16_Cloudflareトンネルで公開.md) のクイックトンネル（一時URL）から、**自分のドメインで固定URL・自動起動**する **名前付きトンネル**へ進めます。あわせて、**公開運用で必要になる設定**（セッション維持・セキュアCookie・秘密情報の扱い）を最初から正しく入れて、よくあるトラブルを未然に防ぎます。前提環境は 13〜16章と同じです。
>
> ドメインやパスワードは環境固有です。本章では例として `shift.example.com` と書きます（[13章の置換早見表](13_本番運用.md) と同じ要領で自分の値に読み替えてください）。

---

## 17-1. 名前付きトンネルにする（固定URL・自動起動）

クイックトンネルは「起動ごとにURLが変わる・プロセスを止めると公開も止まる」一時的なものでした。**名前付きトンネル**にすると、自分のドメインで固定URL、かつ `cloudflared` をサービスとして常駐できます。

手順の概要（Cloudflareの管理画面で進めるのが簡単）:

1. Cloudflareアカウントを作り、**ドメインを追加**（ネームサーバをCloudflareに向ける）。
2. Cloudflare の **Zero Trust → Networks → Tunnels** で新規トンネルを作成。
3. 表示される **インストールコマンド（トークン付き）** をWSLで実行すると、`cloudflared` が**サービスとして常駐**する。実体は次のような形:
   ```
   /usr/bin/cloudflared --no-autoupdate tunnel run --token <長いトークン>
   ```
   `systemctl status cloudflared` で `active (running)` を確認できる。
4. トンネルの **Public Hostname** で `shift.example.com → http://localhost:80`（Nginx）を対応づける。

これで `https://shift.example.com` が固定の公開URLになり、PC起動時に自動でトンネルが張られます。

> 🔰 ポイントは2つ。①公開先は **Nginx(80)** に向ける（静的配信＋Gunicorn中継のため）。②`cloudflared` が常駐サービスなので、`shiftmanager`（アプリ）と独立して動く。

## 17-2. Django 側の設定（ドメイン版）

16章の環境変数を、`.trycloudflare.com` から**自分のドメイン**に置き換えます。systemd の drop-in で渡します。

```
$ printf '%s\n' '[Service]' \
    'Environment=DJANGO_BEHIND_TLS_PROXY=1' \
    'Environment=DJANGO_CSRF_TRUSTED_ORIGINS=https://shift.example.com' \
  | sudo tee /etc/systemd/system/shiftmanager.service.d/tunnel.conf >/dev/null
$ sudo systemctl daemon-reload && sudo systemctl restart shiftmanager
```

- `DJANGO_BEHIND_TLS_PROXY=1` … Cloudflare が付ける `X-Forwarded-Proto: https` を見て、Django に「これはHTTPS」と伝える（`SECURE_PROXY_SSL_HEADER`）。
- `DJANGO_CSRF_TRUSTED_ORIGINS=https://shift.example.com` … ログイン等のPOSTで、ブラウザのオリジンを信頼するために必須。

### 🔰 `ALLOWED_HOSTS` にドメインが無くても動く理由

`ALLOWED_HOSTS` には `localhost` だけでも動きます。**Cloudflare のトンネルは内部へ中継するとき `Host` ヘッダを `localhost`（中継先）にする**ため、Djangoが見る `Host` は `localhost`＝許可済みになるからです。一方ブラウザ側のオリジンは `https://shift.example.com` なので、**POST（CSRF）のために `CSRF_TRUSTED_ORIGINS` にはドメインが必要**、という非対称になります。

> ⚠️ 中継時に元のHostを保持する設定にしている場合は、`ALLOWED_HOSTS` にドメイン（`shift.example.com`）を足してください。`400 (DisallowedHost)` が出たらここを疑います。

## 17-3. セッションを維持する（SECRET_KEY は全ワーカー共通）

公開すると複数ワーカー＋多数アクセスになります。このとき **全ワーカーが同じ `SECRET_KEY` を持っていないと、利用者が勝手にログアウト**します（[13章 13-6](13_本番運用.md) で、英数字の固定鍵を systemd に設定済みなので、本来この問題は起きません）。

公開前に、念のため**全ワーカーに鍵が行き渡っているか**だけ確認しておきます（値は出さず、設定有無のみ）。

```
$ for pid in $(pgrep -f "gunicorn shift_manager"); do
    tr '\0' '\n' < /proc/$pid/environ | grep -c '^DJANGO_SECRET_KEY='
  done
```

すべて `1` なら全員に鍵が渡っています。`0` が混ざる／鍵が無い場合は 13-6 を見直してください（多くは **鍵に `%`・`$` が入って systemd に弾かれた**ケース。英数字の鍵にすれば解消）。

## 17-4. セキュアCookie を有効化する（HTTPS一本化が前提）

公開を Cloudflare の HTTPS に一本化するなら、Cookie を **HTTPS でのみ送る**設定にして盗聴を防ぎます。`settings.py` は環境変数で切り替えます。

```python
if os.environ.get('DJANGO_SECURE_COOKIES') == '1':
    SESSION_COOKIE_SECURE = True
    CSRF_COOKIE_SECURE = True
```

drop-in に追加します（綴りに注意）。

```
$ printf 'Environment=DJANGO_SECURE_COOKIES=1\n' \
    | sudo tee -a /etc/systemd/system/shiftmanager.service.d/tunnel.conf >/dev/null
$ sudo systemctl daemon-reload && sudo systemctl restart shiftmanager
# 反映と綴りを確認
$ systemctl show shiftmanager -p Environment | tr ' ' '\n' | grep SECURE_COOKIES
```

> ⚠️ 2つの落とし穴に注意。
> - **環境変数名のタイプミス**：`DJANGO_SECURE_COOKIES` を打ち間違えると黙って無効になる（エラーは出ない）。上の `systemctl show ... | grep` で綴りを必ず確認。
> - **HTTP では使えない**：セキュアCookieを有効にすると、`http://192.168.10.8/`（LANのHTTP）では Cookie が送られず**ログインが維持できません**。**公開をHTTPSに一本化する前提**で有効化します。HTTP併用を続けるなら有効化しない。
>
> 🔰 セキュアCookieが正しく効くのは、17-2 の `DJANGO_BEHIND_TLS_PROXY=1` で「Cloudflare経由＝HTTPS」とDjangoが認識できているからです（セットで設定）。

## 17-5. 秘密情報（SECRET_KEY / DBパスワード）の安全な扱い

公開環境では秘密情報の取り回しが重要です。原則は3つ。

- **Git に入れない**：`SECRET_KEY`・`DB_PASSWORD` はコード/リポジトリに載せず、**systemd の drop-in（root保護）**にだけ置く（[10章](10_GitHubで管理.md)）。
- **ファイルを保護**：秘密を含む drop-in は `sudo chmod 640`（rootのみ読める）にする。
- **生成・更新時は値を画面に出さない**：シェル変数に入れて使い、ターミナルに表示しない。

### DBパスワードを更新（ローテーション）する

万一パスワードが漏れた（ログに出た等）ときは、**MySQL側とアプリ設定を同じ新パスワードで**、値を画面に出さずに更新します。

```
$ NEWPW=$(python3 -c "import secrets,string; print(''.join(secrets.choice(string.ascii_letters+string.digits) for _ in range(20)))")
$ sudo mysql -e "ALTER USER 'shift'@'localhost' IDENTIFIED BY '$NEWPW'; FLUSH PRIVILEGES;"
$ printf '[Service]\nEnvironment=DJANGO_DB=mysql\nEnvironment=DB_PASSWORD=%s\n' "$NEWPW" \
    | sudo tee /etc/systemd/system/shiftmanager.service.d/mysql.conf >/dev/null
$ sudo chmod 640 /etc/systemd/system/shiftmanager.service.d/mysql.conf
$ sudo systemctl daemon-reload && sudo systemctl restart shiftmanager
```

`NEWPW` はシェル変数の中だけにあり表示されません。MySQLとアプリの両方に同じ値を入れるので、再起動後そのまま接続できます。

確認（誤ログインを試し、**500=接続失敗ではなく 200=ログイン失敗メッセージ**ならDB接続OK）:

```
$ curl -s -o /dev/null -w "%{http_code}\n" http://localhost/login/   # 200系ならDBは生きている
```

## 17-6. 公開運用チェックリスト（つまずき予防）

公開後に問題が出たら、ここを上から確認します。

| 症状 | 原因 | 対処 |
| --- | --- | --- |
| **何度もログアウトする** | ワーカー間で `SECRET_KEY` 不一致 | 13-6 で英数字の固定鍵を設定（17-3で確認）。`%`/`$` を含む鍵は systemd に弾かれる |
| **`400 Bad Request`** | `ALLOWED_HOSTS` に公開ホストが無い | 中継Hostに合わせて `localhost` か `ドメイン` を許可（17-2） |
| **ログインで `403 (CSRF)`** | `CSRF_TRUSTED_ORIGINS`/HTTPS判定の不足 | `DJANGO_CSRF_TRUSTED_ORIGINS=https://ドメイン` ＋ `DJANGO_BEHIND_TLS_PROXY=1`（17-2） |
| **セキュアCookieが効かない** | 環境変数名のタイプミス | `systemctl show ... | grep SECURE_COOKIES` で綴り確認（17-4） |
| **HTTPでログインできない** | セキュアCookie有効＝HTTPS限定 | HTTPS一本化前提で使う。HTTP併用なら無効化（17-4） |
| **全ページ 500** | `DB_PASSWORD` 誤り／MySQL未起動 | 17-5の手順で再設定、`systemctl status mysql` 確認 |

## この章のまとめ

- 名前付きトンネルで**固定URL・自動起動**の恒久公開にした。
- 公開運用に必要な設定を**最初から正しく**入れた：セッション維持（全ワーカー共通の `SECRET_KEY`／13章）・HTTPS判定とCSRF（17-2）・セキュアCookie（17-4）・秘密情報の保護（17-5）。
- 起こりがちな症状は **17-6 のチェックリスト**で素早く切り分けられる。

**要点**：公開して複数ワーカー・多数アクセスになると、開発では見えなかった点（鍵の共有・Cookieのscheme・秘密情報の取り回し）が効いてくる。秘密情報は **systemd 側に固定値で・root保護で・Git外で** 持つのが基本です。
