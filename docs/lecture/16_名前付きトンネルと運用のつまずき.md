# 16章 名前付きトンネルでの恒久公開と、運用でのつまずき

> 🔰 この章も **運用編** です。[15章](15_Cloudflareトンネルで公開.md) のクイックトンネル（一時URL）から、**自分のドメインで固定URL・自動起動**する **名前付きトンネル**へ進めます。さらに、実際に公開運用して遭遇した **3つのつまずき**（何度もログアウトする／セキュアCookie／DBパスワードの更新）と、その直し方を記録します。前提環境は 12〜15章と同じです。
>
> ドメインやパスワードは環境固有です。本章では例として `shift.example.com` と書きます（[12章の置換早見表](12_本番運用.md) と同じ要領で自分の値に読み替えてください）。

## 16-1. 名前付きトンネルにする（固定URL・自動起動）

クイックトンネルは「起動ごとにURLが変わる・プロセスを止めると公開も止まる」一時的なものでした。**名前付きトンネル**にすると、自分のドメインで固定URL、かつ `cloudflared` をサービスとして常駐できます。

手順の概要（Cloudflareの管理画面で進めるのが簡単）:

1. Cloudflareアカウントを作り、**ドメインを追加**（ネームサーバをCloudflareに向ける）。
2. Cloudflare の **Zero Trust → Networks → Tunnels** で新規トンネルを作成。
3. 表示される **インストールコマンド（トークン付き）** をWSLで実行すると、`cloudflared` が**サービスとして常駐**する。実体は次のような形:
   ```
   /usr/bin/cloudflared --no-autoupdate tunnel run --token <長いトークン>
   ```
   `systemctl status cloudflared` で `active (running)` を確認できる。
4. トンネルの **Public Hostname** で `shift.example.com → http://localhost:80`（Nginx）を対応づける。

これで `https://shift.example.com` が固定の公開URLになり、PC起動時に自動でトンネルが張られます。

> 🔰 ポイントは2つ。①公開先は **Nginx(80)** に向ける（静的配信＋Gunicorn中継のため）。②`cloudflared` が常駐サービスなので、`shiftmanager`（アプリ）と独立して動く。

## 16-2. Django 側の設定（ドメイン版）

15章の環境変数を、`.trycloudflare.com` から**自分のドメイン**に置き換えます。systemd の drop-in で渡します。

```
$ printf '%s\n' '[Service]' \
    'Environment=DJANGO_BEHIND_TLS_PROXY=1' \
    'Environment=DJANGO_CSRF_TRUSTED_ORIGINS=https://shift.example.com' \
  | sudo tee /etc/systemd/system/shiftmanager.service.d/tunnel.conf >/dev/null
$ sudo systemctl daemon-reload && sudo systemctl restart shiftmanager
```

### 🔰 `ALLOWED_HOSTS` にドメインが無くても動く理由

実は `ALLOWED_HOSTS` には `localhost` だけでも動きます。**Cloudflare のトンネルは、内部へ中継するとき `Host` ヘッダを `localhost`（中継先のホスト）にする**ため、Djangoが見る `Host` は `localhost`＝許可済みになるからです。一方ブラウザ側のオリジンは `https://shift.example.com` なので、**POST（CSRF）のために `CSRF_TRUSTED_ORIGINS` にはドメインが必要**、という非対称になります。

> ⚠️ 中継時に元のHostを保持する設定にしている場合は、`ALLOWED_HOSTS` にドメイン（`shift.example.com`）を足す必要があります。400 (DisallowedHost) が出たらここを疑う。

## 16-3. つまずき① 「何度も勝手にログアウトする」

公開後、**ページを移動するたびにログアウトされる**症状が出ました。

### 原因：ワーカーごとに SECRET_KEY が違う

`settings.py` は鍵をこう読みます。

```python
SECRET_KEY = os.environ.get('DJANGO_SECRET_KEY', get_random_secret_key())
```

`DJANGO_SECRET_KEY` が**環境変数に無い**と、各プロセスが起動時に `get_random_secret_key()` で**別々のランダム鍵**を作ります。Gunicorn は `--workers 3` で複数ワーカーがいるため、**セッションCookieに署名した鍵と、検証するワーカーの鍵が食い違い**、別ワーカーに当たるたびにセッションが無効化＝ログアウトされていました。

### なぜ鍵が「無い」状態に？ — systemd の落とし穴

ユニットには `Environment=DJANGO_SECRET_KEY=...` を書いていたのに、ワーカーには渡っていませんでした。原因は **systemd の `Environment=` が `%` と `$` を特別扱い**すること。`get_random_secret_key()` が生成する鍵には `%` や `$` が混じることがあり、その行が無効化されて**変数が設定されない**状態になっていました。

### 診断：全ワーカーの鍵が一致しているか

`/proc/<pid>/environ` を見て、鍵の md5 を比べます（値は出さない）。

```
$ for pid in $(pgrep -f "gunicorn shift_manager"); do
    line=$(tr '\0' '\n' < /proc/$pid/environ | grep '^DJANGO_SECRET_KEY=')
    [ -z "$line" ] && echo "$pid: 未設定" || echo "$pid: $(printf '%s' "$line" | md5sum | cut -c1-8)"
  done
```

全部「未設定」や md5 がバラバラなら、これが原因です。

### 修正：英数字のみの固定鍵を drop-in で全ワーカーに共有

systemd が確実に扱える **英数字だけの鍵**を生成し、drop-in で設定します（鍵は端末内で生成し、画面に出さない）。

```
$ KEY=$(python3 -c "import secrets,string; print(''.join(secrets.choice(string.ascii_letters+string.digits) for _ in range(64)))")
$ printf '[Service]\nEnvironment=DJANGO_SECRET_KEY=%s\n' "$KEY" \
    | sudo tee /etc/systemd/system/shiftmanager.service.d/secret.conf >/dev/null
$ sudo chmod 640 /etc/systemd/system/shiftmanager.service.d/secret.conf
$ sudo systemctl daemon-reload && sudo systemctl restart shiftmanager
```

drop-in は元のユニットより後に読まれるので、壊れていた鍵行を上書きします。再起動後は全ワーカーが**同じ鍵**になり、ログアウトしなくなりました。

> 🔰 別解：`%`/`$` を含む鍵を使いたいなら、`EnvironmentFile=/etc/シークレット.env`（値をそのまま読む＝特殊文字の影響を受けない）に書く方法もあります。秘密情報の固定値はこの方式が堅実です。

## 16-4. つまずき② セキュアCookie と Host の扱い

公開がHTTPS（Cloudflare）一本なら、Cookie を **HTTPS限定**にすると安全です。`settings.py` は次の環境変数で切り替えます。

```python
if os.environ.get('DJANGO_SECURE_COOKIES') == '1':
    SESSION_COOKIE_SECURE = True
    CSRF_COOKIE_SECURE = True
```

このとき遭遇した小さなミス2つ:

- **環境変数名のタイプミス**（`DJANGO_SECURE_COKKIES` のように打ち間違えると黙って無効）。`systemctl show shiftmanager -p Environment` で綴りを確認する。
- **セキュアCookieを有効にすると、HTTPでのログインが維持できなくなる**。`http://192.168.10.8/`（LANのHTTP）ではCookieが送られず、ログイン状態が保てません。**公開をCloudflareのHTTPSに一本化する前提**でのみ有効化します。

> 🔰 Cloudflare 経由だと `X-Forwarded-Proto: https` が来るので、`DJANGO_BEHIND_TLS_PROXY=1`（＝`SECURE_PROXY_SSL_HEADER`）と組み合わせて初めて、Djangoが「これはHTTPS」と判断してセキュアCookieを正しく扱えます。

## 16-5. つまずき③ DBパスワードを安全に更新する

設定の確認中に **DBパスワードを画面に出してしまった**ため、ローテーション（変更）しました。手順は「MySQL側のパスワード変更」と「アプリ設定の更新」を**同じ新パスワードで**行い、かつ**値を画面に出さない**のがコツです。

```
$ NEWPW=$(python3 -c "import secrets,string; print(''.join(secrets.choice(string.ascii_letters+string.digits) for _ in range(20)))")
$ sudo mysql -e "ALTER USER 'shift'@'localhost' IDENTIFIED BY '$NEWPW'; FLUSH PRIVILEGES;"
$ printf '[Service]\nEnvironment=DJANGO_DB=mysql\nEnvironment=DB_PASSWORD=%s\n' "$NEWPW" \
    | sudo tee /etc/systemd/system/shiftmanager.service.d/mysql.conf >/dev/null
$ sudo chmod 640 /etc/systemd/system/shiftmanager.service.d/mysql.conf
$ sudo systemctl daemon-reload && sudo systemctl restart shiftmanager
```

`NEWPW` はシェル変数の中だけにあり、ターミナルに表示されません。MySQLとアプリの両方に同じ値を入れるので、再起動後そのまま接続できます。

### 確認：新パスワードで接続できているか

誤ったパスワードでログインを試し、**500（接続失敗）ではなく 200（ログイン失敗メッセージ）** が返れば、DBへの問い合わせ＝接続は成功しています。

```
$ curl -s -o /dev/null -w "%{http_code}\n" -X POST http://localhost/login/   # CSRF等は別途。200系ならDBは生きている
```

> ⚠️ 秘密情報を含む drop-in は `chmod 640`（root専用）にします。**`SECRET_KEY`・`DB_PASSWORD` は Git に絶対入れない**（[10章](10_GitHubで管理.md)）。これらは systemd 側（root保護）にだけ置きます。
