# 9章 権限と提出ルール（発展機能）

8章までで動くアプリを、ここからさらに**実運用に耐える形**へ育てます。この章では3つの拡張を、すべて**既存DBを壊さずマイグレーションで**入れていきます。

1. **権限を3段階に**（システム管理者／リーダー／クルー）。
2. **締切後ルールの選択**（締切後は提出・編集をどう扱うか）。
3. **期間の表示／非表示**（クルーに見せる前に準備する）。

> 🔰 発展編はコアの差分として説明します。各拡張は「モデルを変える → マイグレーション → ビュー/テンプレートを直す」の流れ。8章までの「モデルを変えたら makemigrations → migrate」をここでも繰り返します。

---

## 9-1. 権限を3段階にする

これまで「リーダー／クルー」の2段階でした。これを次の3段階にします。

| ロール | 権限 |
| --- | --- |
| システム管理者（admin） | すべて。**アカウント管理も可能** |
| リーダー（leader） | アカウント管理**以外**は管理者と同じ（期間設定・提出状況・確定シフト） |
| クルー（crew） | 提出・閲覧のみ |

### モデルを書き換える

`Role` に `ADMIN` を足し、`STAFF` を `CREW`（クルー）に名称変更します。権限はプロパティで表現します。

ファイル: `accounts/models.py`

```python
from django.contrib.auth.models import AbstractUser
from django.db import models


class User(AbstractUser):
    class Role(models.TextChoices):
        ADMIN = "admin", "システム管理者"
        LEADER = "leader", "リーダー"
        CREW = "crew", "クルー"

    name = models.CharField("表示名", max_length=100)
    role = models.CharField(
        "ロール", max_length=16, choices=Role.choices, default=Role.CREW
    )

    class Meta:
        verbose_name = "ユーザー"
        verbose_name_plural = "ユーザー"

    def __str__(self):
        return f"{self.name}（{self.username}）"

    @property
    def is_admin(self) -> bool:
        return self.role == self.Role.ADMIN

    @property
    def is_leader(self) -> bool:
        return self.role == self.Role.LEADER

    @property
    def can_manage(self) -> bool:
        """期間・提出・確定シフトの管理ができるか（管理者 or リーダー）。"""
        return self.role in (self.Role.ADMIN, self.Role.LEADER)

    @property
    def can_manage_accounts(self) -> bool:
        """アカウント管理ができるか（システム管理者のみ）。"""
        return self.role == self.Role.ADMIN
```

**解説**

- 権限判定を**プロパティに集約**しました。画面やビューでは `user.can_manage` / `user.can_manage_accounts` と書くだけ。ロールが増減しても、判定ロジックはこの1か所で済みます。
- これまでの `is_leader`（リーダーか）は残しつつ、「管理できるか」は `can_manage` に置き換えていきます。

### データ移行（既存ユーザーのロールを付け替える）

`role` の中身を `staff → crew`、既存のリーダー（および superuser）を `admin` に寄せます。**スキーマは変わらない（選択肢が変わるだけ）**ので、ここは**データマイグレーション**で値を書き換えます。

まず空のマイグレーションを作ります。

```
$ python manage.py makemigrations accounts --empty --name remap_roles
```

生成された `accounts/migrations/0002_remap_roles.py`（番号は環境による）を次のように編集します。

```python
from django.db import migrations


def remap_forward(apps, schema_editor):
    User = apps.get_model("accounts", "User")
    # 旧クルー staff → crew
    User.objects.filter(role="staff").update(role="crew")
    # 既存リーダー、または superuser はシステム管理者へ寄せる
    User.objects.filter(role="leader", is_superuser=True).update(role="admin")


def remap_backward(apps, schema_editor):
    User = apps.get_model("accounts", "User")
    User.objects.filter(role="crew").update(role="staff")
    User.objects.filter(role="admin").update(role="leader")


class Migration(migrations.Migration):
    dependencies = [
        ("accounts", "0001_initial"),  # 直前のマイグレーション名に合わせる
    ]
    operations = [
        migrations.AlterField(  # 選択肢(choices)の変更を反映
            model_name="user",
            name="role",
            field=__import__("django.db.models", fromlist=["CharField"]).CharField(
                default="crew", max_length=16, verbose_name="ロール",
                choices=[("admin", "システム管理者"), ("leader", "リーダー"), ("crew", "クルー")],
            ),
        ),
        migrations.RunPython(remap_forward, remap_backward),
    ]
```

> 🔰 実際には `makemigrations` を普通に実行すれば `AlterField`（choices変更）部分は自動生成されます。そこに `RunPython(...)` を**手で足す**のが現実的です。上の例は「何が起きているか」を1ファイルで見せるために両方を明示しています。`apps.get_model` で**その時点のモデル**を取るのがデータ移行の作法（最新のモデルクラスを直接importしない）。

```
$ python manage.py migrate
```

> 💡 少人数なら、移行を作らず /admin で手作業でロールを付け替えてもかまいません。ただし「環境を再現できる」「やり直せる」のがマイグレーションの価値です。

### デコレータを2種類に

3章の `leader_required` を、用途別の2つに置き換えます。

ファイル: `accounts/decorators.py`

```python
from functools import wraps

from django.contrib import messages
from django.shortcuts import redirect


def manager_required(view):
    """ログイン済み かつ 管理権限（管理者 または リーダー）のみ許可。"""

    @wraps(view)
    def wrapper(request, *args, **kwargs):
        if not request.user.is_authenticated:
            return redirect("login")
        if not request.user.can_manage:
            messages.error(request, "この画面は管理権限が必要です。")
            return redirect("home")
        return view(request, *args, **kwargs)

    return wrapper


def admin_required(view):
    """ログイン済み かつ システム管理者のみ許可（アカウント管理用）。"""

    @wraps(view)
    def wrapper(request, *args, **kwargs):
        if not request.user.is_authenticated:
            return redirect("login")
        if not request.user.can_manage_accounts:
            messages.error(request, "この画面はシステム管理者のみ利用できます。")
            return redirect("home")
        return view(request, *args, **kwargs)

    return wrapper
```

### 既存コードの置き換え

これまで `leader_required` / `is_leader` / `Role.STAFF` を使っていた箇所を、役割に応じて置き換えます。

- **期間・提出状況・確定シフトの管理系ビュー**: `@leader_required` → `@manager_required`（リーダーも管理者も可）。
- **アカウント管理系ビュー**（8章の `account_*`）: `@leader_required` → `@admin_required`（管理者のみ）。
- **テンプレートの分岐**: `user.is_leader` → `user.can_manage`（`base.html` / `home.html` / `confirmed.html` / `manage_top.html` など）。
- **クルー一覧のフィルタ**: `role=User.Role.STAFF` → `role=User.Role.CREW`（`period_status` / `manage_top`）。
- **ホーム・提出・履歴の権限判定**: `request.user.is_leader` → `request.user.can_manage`。

`manage_top` は「アカウント管理カードは管理者だけに見せる」よう、テンプレートを少し変えます。

ファイル: `shifts/templates/shifts/manage_top.html`（アカウント管理カードを `can_manage_accounts` で囲む）

```html
  {% if user.can_manage_accounts %}
  <div class="col-12 col-md-6">
    <div class="card h-100 shadow-sm">
      <div class="card-body">
        <h2 class="h6">アカウント管理</h2>
        <p class="text-muted small mb-2">クルー等の作成・編集・削除・パスワード再発行・有効/無効化</p>
        <p class="small mb-2">有効なクルー: {{ staff_count }} 名</p>
        <a class="btn btn-primary btn-sm" href="{% url 'manage_accounts' %}">アカウント管理へ</a>
      </div>
    </div>
  </div>
  {% endif %}
```

> ⚠️ 置き換えは**ビュー側（デコレータ・判定）とテンプレート側（表示制御）の両方**を必ずやること。片方だけだと「ボタンは消えたが直接URLを叩けば実行できる」穴が残ります。

## 9-2. 締切後ルールを選べるようにする

これまで「締切後は一律で閲覧のみ」でした。期間ごとに次の3つから選べるようにします。

- **全員 提出・編集不可**（`lock_all`）
- **未提出者のみ提出可・編集は不可**（`late_submit`）
- **締切後も編集可**（`editable`）。任意で「編集最終期限」を設定し、それまで可。空なら手動で締め切るまで可。

### モデルにフィールドを追加

ファイル: `shifts/models.py`（`ShiftPeriod` 内）

```python
class ShiftPeriod(models.Model):
    class Status(models.TextChoices):
        OPEN = "open", "募集中"
        CLOSED = "closed", "締切後"

    class PostDeadlinePolicy(models.TextChoices):
        LOCK_ALL = "lock_all", "全員 提出・編集不可"
        LATE_SUBMIT = "late_submit", "未提出者のみ提出可（編集不可）"
        EDITABLE = "editable", "締切後も編集可（編集最終期限まで）"

    # ... 既存フィールド（title 〜 status, created_by ...）はそのまま ...

    post_deadline_policy = models.CharField(
        "締切後の扱い", max_length=16,
        choices=PostDeadlinePolicy.choices, default=PostDeadlinePolicy.LOCK_ALL,
    )
    edit_deadline = models.DateTimeField("編集最終期限", null=True, blank=True)
```

マイグレーション:

```
$ python manage.py makemigrations
$ python manage.py migrate
```

> 🔰 新フィールドに `default` / `null=True` があるので、既存行があっても安全に追加できます（既存期間はすべて `lock_all`＝従来どおりになります）。

### 提出可否の判定ロジック

コアの「`read_only = is_closed`」を、ポリシーを踏まえた関数に置き換えます。

ファイル: `shifts/views.py`（ヘルパーを追加）

```python
def _manually_closed(period, now) -> bool:
    """締切日時より前なのに、管理者が手動で締め切った状態か。"""
    return period.status == ShiftPeriod.Status.CLOSED and period.deadline >= now


def _submission_permission(period, now, submitted):
    """締切後ポリシーを踏まえ、(新規提出できるか, 既存を編集できるか) を返す。"""
    Policy = ShiftPeriod.PostDeadlinePolicy

    # 手動締切（status=closed）は常に全面クローズ
    if period.status == ShiftPeriod.Status.CLOSED:
        return (False, False)
    # 締切前は誰でも提出・編集可能
    if now < period.deadline:
        return (True, True)

    # ここから締切後（締切日時を過ぎ、手動締切はされていない）
    policy = period.post_deadline_policy
    if policy == Policy.LOCK_ALL:
        return (False, False)
    if policy == Policy.LATE_SUBMIT:
        return (not submitted, False)  # 未提出者のみ提出可・編集は不可
    if policy == Policy.EDITABLE:
        if period.edit_deadline:  # 編集最終期限まで
            return (now < period.edit_deadline, now < period.edit_deadline)
        return (True, True)  # 手動で締め切るまで可
    return (False, False)
```

### ビューに組み込む

### 案内メッセージ（任意）

締切後でも提出/編集できるケースでは、その旨を案内すると親切です。提出画面のビューに次のヘルパーを足し、`context` に `"notice": _submit_notice(period, now, submitted, read_only)` を渡します。

```python
def _submit_notice(period, now, submitted, read_only):
    """提出画面に出す案内（(bootstrapレベル, 文言) のタプル）。無ければ None。"""
    Policy = ShiftPeriod.PostDeadlinePolicy
    past_deadline = now >= period.deadline and period.status != ShiftPeriod.Status.CLOSED

    if read_only:
        return None  # 閲覧のみの案内は既存の表示に任せる
    if past_deadline:
        if period.post_deadline_policy == Policy.EDITABLE:
            if period.edit_deadline:
                limit = timezone.localtime(period.edit_deadline).strftime("%Y/%m/%d %H:%M")
                return ("info", f"締切後ですが、編集最終期限（{limit}）まで編集できます。")
            return ("info", "締切後ですが、管理者が締め切るまで編集できます。")
        if period.post_deadline_policy == Policy.LATE_SUBMIT:
            return ("info", "締切を過ぎていますが、未提出のため提出できます（提出後の編集は不可）。")
    return None
```

```html
  <div class="alert alert-{{ notice.0 }}">{{ notice.1 }}</div>
{% endif %}
```

## 9-3. 期間の表示／非表示

「準備中の期間はクルーに見せず、整ったら公開する」を実現します。

### モデルにフラグ追加

ファイル: `shifts/models.py`（`ShiftPeriod` 内）

```python
    is_visible = models.BooleanField("クルーに表示", default=True)
```

```
$ python manage.py makemigrations
$ python manage.py migrate
```

### ホームでフィルタ

クルーには非表示の期間を出しません。リーダー/管理者は確認のため全件見られます。

ファイル: `shifts/views.py`（`home`）

```python
    periods = list(ShiftPeriod.objects.filter(is_visible=True)) \
        if not request.user.can_manage else list(ShiftPeriod.objects.all())
```

> 🔰 上は説明用に三項で書いています。実コードでは「管理者は全件、クルーは可視のみ」を `if request.user.can_manage:` の分岐内でそれぞれ取得しても構いません（読みやすさ優先）。

提出ビュー（`shift_submit`）にも、非表示期間へのクルーの直接アクセスを禁止するガードを入れます。

```python
    if not period.is_visible and not request.user.can_manage:
        messages.error(request, "この期間は現在表示されていません。")
        return redirect("home")
```

### 表示/非表示トグルのビューとURL

ファイル: `shifts/views.py`

```python
@manager_required
def manage_period_toggle_visible(request, pk):
    """クルーへの表示/非表示を切り替える。"""
    period = get_object_or_404(ShiftPeriod, pk=pk)
    if request.method == "POST":
        period.is_visible = not period.is_visible
        period.save(update_fields=["is_visible"])
        state = "表示" if period.is_visible else "非表示"
        messages.success(request, f"「{period}」をクルーに{state}にしました。")
    return redirect("manage_periods")
```

ファイル: `shifts/urls.py`（追記）

```python
    path("manage/periods/<int:pk>/visibility/", views.manage_period_toggle_visible, name="manage_period_toggle_visible"),
```

### 一覧にトグルスイッチの列を足す

8章の `manage_periods.html` の表ヘッダに `<th class="text-center">クルー表示</th>` を追加し、各行に次のセルを足します（Bootstrapのトグルスイッチ。変更で即送信）。


## 9-4. 動作確認

### ✅ 権限

1. 管理者・リーダー・クルーを1人ずつ用意。
2. **リーダー**でログイン → 管理メニューに「アカウント管理」カードが**出ない**。アカウント管理URL（`/manage/accounts/`）を直接開くと「システム管理者のみ」で弾かれる。期間設定・提出状況は使える。
3. **管理者**でログイン → すべて使える。

### ✅ 締切後ルール

1. 期間を作り、締切日時を過去にし、`post_deadline_policy` を切り替えて試す。
2. `lock_all`: 締切後は全員閲覧のみ。
3. `late_submit`: 未提出のクルーは提出できる（提出後は編集不可）。提出済みは閲覧のみ。
4. `editable` ＋ 編集最終期限を未来に: 締切後も編集でき、案内メッセージが出る。期限を過ぎると閲覧のみに。

### ✅ 表示/非表示

1. リーダーで期間のスイッチを「非表示」に → クルーのホームから消える。直接URLを開いても弾かれる。
2. リーダー/管理者の管理画面・ホームには引き続き出る。

## 9-5. つまずきポイント

- **migrate でロールが消えた/おかしい** → データ移行の `RunPython` の対象 `role` 値（`staff`/`leader`）が実データと違う。/admin で確認してから移行。
- **テンプレートで権限が効かない** → `is_leader` の置き換え漏れ。`grep` で `is_leader` を検索して洗い出す。
- **締切後に提出できてしまう/できない** → `_submission_permission` の戻り値（新規可, 編集可）の使い分けミス。提出済みは編集可否、未提出は新規可否で判定。
- **非表示にしたのにクルーに見える** → `home` のフィルタ漏れ、または `shift_submit` のガード漏れ。
