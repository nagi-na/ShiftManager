# 終章 ― 全体のまとめ

おつかれさまでした。要件定義から始め、**動くアプリ**を作り（開発編）、それを**実運用構成へ載せ**（本番編）、最後に **MySQL へ移行**するところまで、ひとつのWebアプリを一気通貫でやり切りました。この章で全体を振り返ります。

---

## 本番編（12〜13章）でやったこと

開発編で完成したアプリを、外部公開に耐える構成へ載せ替えました。

| 章 | テーマ | 主な作業 |
| --- | --- | --- |
| 12 | 本番運用 | `DEBUG`/`ALLOWED_HOSTS`/`SECRET_KEY` を環境変数化、`collectstatic`、**Gunicorn**でアプリ起動、**Nginx**を前段に（静的配信＋中継）、**systemd**で常駐化、`check --deploy`で点検 |
| 13 | MySQLの導入（運用編） | 稼働中の本番を止めずに、`dumpdata`→`migrate`→`loaddata`で**データを保全**しながら SQLite → **MySQL** へ移行、drop-inで切替、ロールバック手順も用意 |
| 14 | LANへの公開（運用編） | WSLの**ミラーモード**でホストと同じLANに載せ、`ALLOWED_HOSTS`にLAN IPを追加、Windowsファイアウォールで許可し、他端末から接続可能に |
| 15 | Cloudflareトンネルで公開（運用編） | `cloudflared` で外向きトンネルを張り、ポート開放/CGNAT不要で**インターネットから正規HTTPS公開**。`ALLOWED_HOSTS`/`CSRF_TRUSTED_ORIGINS`/プロキシHTTPS判定を設定 |

本番編の核心は「**開発に最適な構成**」と「**公開・常時稼働に耐える構成**」は別物だ、という理解です。

## 開発環境と本番環境 ― 構成の違い

同じアプリでも、動かす環境でこれだけ構成が変わります。これが本番編の総まとめです。

| 項目 | 開発環境 | 本番環境 | なぜ違うか |
| --- | --- | --- | --- |
| アプリの起動 | `runserver` | **Gunicorn**（WSGIサーバー, 複数ワーカー） | runserverは開発専用。同時処理・常時稼働に非対応 |
| 前段サーバー | なし | **Nginx**（リバースプロキシ） | 静的配信・接続の受け口・将来のHTTPS化を担う |
| `DEBUG` | `True` | **`False`** | Trueはエラーで内部情報が露出する |
| `SECRET_KEY` | 自動生成（使い捨て） | **環境変数で秘匿**（固定の長い鍵） | 漏れると署名を偽造されうる。Gitに載せない |
| `ALLOWED_HOSTS` | 空でよい | **公開ホスト名/IPを明示** | なりすまし(Host)対策。未設定だと400 |
| 静的ファイル | Djangoが自動配信 | `collectstatic`→**Nginxが配信** | DEBUG=FalseのDjangoは静的を配信しない |
| データベース | SQLite（ファイル） | SQLite または **MySQL** | 同時書き込みが増えるとMySQLが有利 |
| 常駐 | 都度手動で起動 | **systemd**（自動起動・自動再起動） | サーバー再起動後も無人で復帰させる |
| 秘密情報の置き場 | （ほぼ不要） | **systemdのEnvironment等**（root保護・Git外） | 認証情報をコード/リポジトリから隔離 |
| 通信 | HTTP（localhost） | HTTP→将来 **HTTPS**（certbot） | 公開時は暗号化が必須級 |

> 🔰 大事なのは設定値そのものより、**「開発は手軽さ優先、本番は安全性・安定性・性能優先」**という考え方です。この軸を持っていれば、別のフレームワークや別のサーバーでも応用が利きます。

## 講座全体の歩み

```
【開発編 0〜11章】  設計 → 実装 → 発展機能 → Git → テスト    （開発環境で「動くアプリ」）
        │
   [間章] 開発編のまとめ
        │
【本番編 12〜15章】  本番設定 → Gunicorn/Nginx/systemd → MySQL → LAN公開 → Cloudflareで公開  （本番環境で「公開できるアプリ」）
        │
   [終章] 全体のまとめ ← いまここ
```

- **開発編**: アルバイトのシフト提出という実課題を、認証・フォーム・集計・ファイルアップロード・権限・テストまで備えた実用アプリに仕上げた。
- **本番編**: それを `DEBUG=False` の本番構成に載せ、Nginx+Gunicorn+systemdで常駐させ、MySQLへ移行し、LAN内、さらにCloudflare Tunnelでインターネットへ正規HTTPS公開した。

セキュリティ（秘密情報の扱い）、品質（自動テスト）、運用（デプロイ・DB移行・ネットワーク公開）という、**実務で必ず問われる観点**を一通り通過しました。

## 用語ミニ集

本番編で出てきた言葉を、ざっくり一言で。

| 用語 | ざっくり説明 |
| --- | --- |
| **WSGI** | PythonのWebアプリとWebサーバーをつなぐ約束事。Djangoは `wsgi.py` を持つ。 |
| **Gunicorn** | その約束事に従い、Djangoアプリを本番で動かすサーバー。`runserver` の本番版。 |
| **Nginx** | 一番前で受けるWebサーバー。静的ファイルを直接配り、それ以外をGunicornへ渡す。 |
| **リバースプロキシ** | 外からのアクセスを受けて、裏のサーバー（Gunicorn）へ中継する役。Nginxの役割。 |
| **静的ファイル / collectstatic** | CSS/JS/画像など。`collectstatic` で1か所に集め、Nginxが配信する。 |
| **systemd** | Linuxの常駐管理。サービスを自動起動・自動再起動する仕組み。 |
| **環境変数** | プログラムの外から渡す設定値。秘密情報（鍵・パスワード）をコードから外に出すのに使う。 |
| **ソケット接続 / TCP接続** | 同一マシン内の近道（`localhost`）と、ネットワーク経由（`127.0.0.1`等）。MySQLは両者を区別する。 |
| **マイグレーション** | モデル定義の変更をDBの表構造に反映する仕組み（`migrate`）。 |
| **dumpdata / loaddata** | DBの中身をJSONに書き出す／読み込む。DB引っ越し（SQLite→MySQL）に使う。 |
| **ロールバック** | 変更を元の状態に戻すこと。本書は「設定1つ＋SQLite残置」で戻せるようにした。 |
