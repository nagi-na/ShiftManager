# 12章 本番運用

完成したアプリを、自宅または大学のサーバーで**実際に公開**するための章です。開発用の `runserver` と SQLite のままでは本番に向かないので、設定を分け、データベースを MySQL に、配信を Gunicorn + Nginx に切り替えます。

> ⚠️ この章は概念と手順の地図です。サーバーのOSやネットワーク事情で細部は変わります。「開発と本番で何が・なぜ違うのか」を理解することを重視してください。最初は SQLite のまま Gunicorn/Nginx だけ試す、でも構いません。

---

## 12-1. 開発と本番の違いを押さえる

| 項目 | 開発（これまで） | 本番 |
| --- | --- | --- |
| 起動 | `runserver` | Gunicorn（WSGIサーバー） |
| 前段 | なし | Nginx（リバースプロキシ／静的配信） |
| `DEBUG` | `True` | **`False`** |
| `SECRET_KEY` | コードに直書き | 環境変数で秘匿 |
| `ALLOWED_HOSTS` | 空 | 公開するホスト名/IP |
| DB | SQLite | MySQL（複数アクセスに強い） |
| 静的ファイル | Djangoが配信 | `collectstatic` → Nginxが配信 |

要点は「**`DEBUG=False` にする**」「**秘密情報をコードから出す**」「**静的/メディアはWebサーバーに任せる**」の3つです。

---

## 12-2. 設定を環境変数で切り替える

`settings.py` を本番/開発で分けず、**環境変数で値を差し替える**方式にします（シンプルで扱いやすい）。

ファイル: `shift_manager/settings.py`（該当箇所を変更）

```python
import os
from pathlib import Path

BASE_DIR = Path(__file__).resolve().parent.parent

# 秘密鍵は環境変数から。無ければ開発用の仮値（本番では必ず設定する）
SECRET_KEY = os.environ.get("DJANGO_SECRET_KEY", "dev-insecure-key-change-me")

# 環境変数 DJANGO_DEBUG が "1" のときだけ True
DEBUG = os.environ.get("DJANGO_DEBUG", "1") == "1"

# 例: "shift.example.com,192.168.1.10"
ALLOWED_HOSTS = os.environ.get("DJANGO_ALLOWED_HOSTS", "").split(",") if not DEBUG else []
```

**解説**

- **`SECRET_KEY`**: セッションやCSRFの署名に使う鍵。漏れると改ざんされうるので、本番では環境変数で渡し、コード/Gitには載せません。
- **`DEBUG=False` が本番の絶対条件**: `True` のままだとエラー画面に**ソースや設定が丸見え**になります。
- **`ALLOWED_HOSTS`**: `DEBUG=False` のとき、ここに無いホスト名でのアクセスは拒否されます。公開するドメイン/IPを入れます。

> 🔰 環境変数は、後述の systemd の `Environment=` や、`.env` ファイル＋`python-dotenv` などで渡せます。秘密情報は**リポジトリに入れない**のが鉄則です。

## 12-3. データベースを MySQL にする

複数人が同時に使う本番では、SQLite より MySQL/PostgreSQL が安心です。ここでは MySQL を例にします。

### ドライバの導入

```
$ pip install mysqlclient
```

> ⚠️ `mysqlclient` のインストールには、OS側に MySQL 開発用ライブラリが必要です。例（Ubuntu/Debian）: `sudo apt install default-libmysqlclient-dev build-essential pkg-config`。

### DB と接続ユーザーを作る（MySQL側）

```sql
CREATE DATABASE shift_manager CHARACTER SET utf8mb4;
CREATE USER 'shift'@'localhost' IDENTIFIED BY '<強いパスワード>';
GRANT ALL PRIVILEGES ON shift_manager.* TO 'shift'@'localhost';
FLUSH PRIVILEGES;
```

- `utf8mb4` … 絵文字も含む完全なUTF-8。日本語運用でも安心。

### settings の DATABASES

ファイル: `shift_manager/settings.py`

```python
if os.environ.get("DJANGO_DB", "sqlite") == "mysql":
    DATABASES = {
        "default": {
            "ENGINE": "django.db.backends.mysql",
            "NAME": os.environ.get("DB_NAME", "shift_manager"),
            "USER": os.environ.get("DB_USER", "shift"),
            "PASSWORD": os.environ.get("DB_PASSWORD", ""),
            "HOST": os.environ.get("DB_HOST", "localhost"),
            "PORT": os.environ.get("DB_PORT", "3306"),
            "OPTIONS": {"charset": "utf8mb4"},
        }
    }
else:
    DATABASES = {
        "default": {
            "ENGINE": "django.db.backends.sqlite3",
            "NAME": BASE_DIR / "db.sqlite3",
        }
    }
```

環境変数 `DJANGO_DB=mysql` のときだけ MySQL を使い、ふだんの開発は SQLite のまま、という切り替えです。

### マイグレーションとデータ

新しい MySQL は空なので、テーブルを作り、管理者を作り直します。

```
$ DJANGO_DB=mysql python manage.py migrate
$ DJANGO_DB=mysql python manage.py createsuperuser
```

> 🔰 SQLiteのデータを移したい場合は `dumpdata` → `loaddata` という手もありますが、本講座では「本番DBは作り直す」前提で十分です。`makemigrations` は不要（モデルは変わっていない）で、`migrate` だけ実行します。

## 12-4. 静的ファイルを集める

本番では Django ではなく Nginx が CSS/JS を配信します。そのために、各所に散らばる静的ファイルを1か所へ集めます。

ファイル: `shift_manager/settings.py`

```python
STATIC_URL = 'static/'
STATICFILES_DIRS = [BASE_DIR / 'static']
STATIC_ROOT = BASE_DIR / 'staticfiles'   # collectstatic の集約先（本番用）
```

```
$ python manage.py collectstatic
```

`staticfiles/` に管理画面などの静的ファイルも含めて集約されます。これを Nginx が配信します。

> 🔰 本アプリの見た目は CDN の Bootstrap なので自前CSSは少ないですが、**Django管理画面（/admin）のCSS**などのために `collectstatic` は本番で必須です。

## 12-5. アプリを Gunicorn で動かす

`runserver` は開発専用。本番は **Gunicorn**（WSGIアプリケーションサーバー）で動かします。

```
$ pip install gunicorn
$ gunicorn shift_manager.wsgi:application --bind 127.0.0.1:8000 --workers 3
```

- `shift_manager.wsgi:application` … プロジェクトに最初からある `wsgi.py` の `application`。
- `--workers 3` … 同時処理のプロセス数（目安: CPUコア数×2＋1）。
- `127.0.0.1:8000` … ローカルだけで待ち受け、外からは直接触らせず Nginx 経由にします。

### systemd で常駐させる

サーバー再起動後も自動で立ち上がるよう、サービス化します。

ファイル: `/etc/systemd/system/shift.service`（例）

```ini
[Unit]
Description=Shift Manager (Gunicorn)
After=network.target

[Service]
User=youruser
WorkingDirectory=/path/to/shift_manager_app
Environment=DJANGO_DEBUG=0
Environment=DJANGO_SECRET_KEY=<本番用の長いランダム文字列>
Environment=DJANGO_ALLOWED_HOSTS=shift.example.com
Environment=DJANGO_DB=mysql
Environment=DB_PASSWORD=<DBパスワード>
ExecStart=/path/to/shift_manager_app/venv/bin/gunicorn \
    shift_manager.wsgi:application --bind 127.0.0.1:8000 --workers 3
Restart=always

[Install]
WantedBy=multi-user.target
```

```
$ sudo systemctl daemon-reload
$ sudo systemctl enable --now shift
$ sudo systemctl status shift
```

> 🔰 秘密情報（`SECRET_KEY` / `DB_PASSWORD`）はこの**サービス定義（root権限で保護）**に置きます。リポジトリには入れません。

## 12-6. Nginx を前段に置く

Nginx が「外からのアクセスの受け口」になり、静的/メディアは自分で配信し、それ以外を Gunicorn へ中継します。

ファイル: `/etc/nginx/sites-available/shift`（例）

```nginx
server {
    listen 80;
    server_name shift.example.com;

    location /static/ {
        alias /path/to/shift_manager_app/staticfiles/;
    }
    location /media/ {
        alias /path/to/shift_manager_app/media/;
    }
    location / {
        proxy_pass http://127.0.0.1:8000;
        proxy_set_header Host $host;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    client_max_body_size 12M;   # 確定シフトPDF(最大10MB)に余裕を持たせる
}
```

```
$ sudo ln -s /etc/nginx/sites-available/shift /etc/nginx/sites-enabled/
$ sudo nginx -t && sudo systemctl reload nginx
```

**解説**

- `/static/` `/media/` は Nginx が直接ファイルを返します（速い・Djangoを煩わせない）。本番では `urls.py` の開発用 `static(...)` 配信（`DEBUG=True` のときだけ有効）は使われません。
- それ以外は `proxy_pass` で Gunicorn へ。`X-Forwarded-*` ヘッダで「本当のアクセス元・スキーム」を Django に伝えます。
- `client_max_body_size` … アップロード上限。フォーム側の10MB制限と整合させます（これが小さいとPDFアップロードが Nginx で弾かれます）。

> 🔰 **HTTPS化**: 公開するなら TLS（https）は必須級です。`certbot`（Let's Encrypt）で証明書を取り、443で配信、80→443へリダイレクトします。併せて settings で `SECURE_SSL_REDIRECT` / `SESSION_COOKIE_SECURE` / `CSRF_COOKIE_SECURE = True` を有効に。`CSRF_TRUSTED_ORIGINS = ["https://shift.example.com"]` も設定します。

## 12-7. 公開前チェック

Djangoには本番設定の点検コマンドがあります。

```
$ DJANGO_DEBUG=0 DJANGO_ALLOWED_HOSTS=shift.example.com python manage.py check --deploy
```

指摘される主な項目:

- `DEBUG=False` になっているか。
- `SECRET_KEY` が安全か（開発用の仮値のままになっていないか）。
- Cookie の Secure 化（HTTPS運用時）。
- `ALLOWED_HOSTS` が設定されているか。

依存ライブラリも固定しておきます。

```
$ pip freeze > requirements.txt
```

別環境では `pip install -r requirements.txt` で同じ構成を再現できます。

## 12-8. デプロイ後の運用メモ

- **コードを更新したら**: `git pull`（使っていれば）→ `pip install -r requirements.txt` → `migrate` → `collectstatic` → `sudo systemctl restart shift`。
- **ログを見る**: `sudo journalctl -u shift -f`（Gunicorn）、Nginx は `/var/log/nginx/`。
- **バックアップ**: MySQL は `mysqldump`、アップロードPDFは `media/` をまるごと。最低限この2つを定期取得。
- **学内サーバーの場合**: 外部公開の可否・ポート開放・ドメインは管理者の規約に従うこと。まずは学内ネットワーク限定での運用が無難です。

---

## 12-9. つまずきポイント

- **`DEBUG=False` にしたら 400 (Bad Request)** → `ALLOWED_HOSTS` にアクセス元のホスト名/IPが入っていない。
- **CSSが当たらない（管理画面が崩れる）** → `collectstatic` 未実行、または Nginx の `/static/` の `alias` パス違い。
- **PDFが開けない/アップロードできない** → Nginx の `/media/` 設定、`client_max_body_size` を確認。
- **`mysqlclient` が入らない** → OSの開発用ライブラリ不足（12-3の `apt install`）。
- **POSTが403 (CSRF)** → HTTPS化したら `CSRF_TRUSTED_ORIGINS` を設定。
