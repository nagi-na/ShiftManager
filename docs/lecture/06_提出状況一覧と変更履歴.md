# 6章 提出状況一覧と変更履歴

リーダーが全員の提出を一目で確認できる **提出状況一覧（S4）** と、提出の **変更履歴** の閲覧画面を作ります。

この章のゴール:

- リーダーが「クルー × 各日」の表で全員の希望を見られる。
- **未提出の人を赤く強調**する。
- 5章で貯めた履歴スナップショットを時系列で閲覧できる。

---

## 6-1. 提出状況一覧のビュー

4章で置いた `period_status` のスタブを本実装に差し替えます。やることは「全クルーを縦、各日を横にした表データを組み立てる」ことです。

ファイル: `shifts/views.py`

まず import に `User` を足します。

```python
from accounts.models import User
```

そして本体:

```python
@leader_required
def period_status(request, pk):
    """S4 提出状況一覧。リーダーが全クルー×各日を表で確認する。"""
    period = get_object_or_404(ShiftPeriod, pk=pk)
    dates = _date_range(period.start_date, period.end_date)

    # 対象は「有効なクルー」全員
    staff_list = User.objects.filter(
        role=User.Role.STAFF, is_active=True
    ).order_by("name", "username")

    # この期間の提出を {user_id: ShiftRequest} で引けるようにする
    requests = {
        r.user_id: r
        for r in ShiftRequest.objects.filter(period=period).prefetch_related("days")
    }

    table = []
    for staff in staff_list:
        req = requests.get(staff.id)
        if req:
            days_map = {d.work_date: d for d in req.days.all()}
            cells = [days_map.get(dt) for dt in dates]
            table.append(
                {"staff": staff, "submitted": True, "cells": cells, "request": req}
            )
        else:
            table.append(
                {
                    "staff": staff,
                    "submitted": False,
                    "cells": [None] * len(dates),
                    "request": None,
                }
            )

    submitted_count = sum(1 for row in table if row["submitted"])
    date_headers = [{"date": dt, "weekday": WEEKDAYS_JP[dt.weekday()]} for dt in dates]

    context = {
        "period": period,
        "date_headers": date_headers,
        "table": table,
        "submitted_count": submitted_count,
        "total_count": len(table),
    }
    return render(request, "shifts/status.html", context)
```

**解説**

- `@leader_required` … この画面はリーダー専用。3章のデコレータをそのまま使います。
- **対象はクルー全員**: `role=User.Role.STAFF` かつ `is_active=True`。退職者（無効）は表に出しません。
- **提出を辞書に**: `{user_id: ShiftRequest}` を1クエリで作り、各クルーの提出を `O(1)` で引けるようにします（ここでもN+1回避）。
- **`prefetch_related("days")`**: 各提出の「各日データ」をまとめて先読み。これが無いと、行ごと・日ごとにクエリが飛びます。
- **セルの組み立て**: 提出済みなら `{日付: 日データ}` を作り、表示する日付の順に並べます。ある日に行が無ければ `None`（空欄）。
- **未提出**: `cells` を `[None] * 日数` にし、`submitted=False`。テンプレートで赤くします。
- `submitted_count` / `total_count` … 「提出 3 / 5 名」のような進捗表示用。

> 🔰 `requests.get(staff.id)` は「無ければ `None` を返す」辞書メソッド。`requests[staff.id]` だとキーが無いとき例外になるので、ここでは `.get` を使います。

**解説**

- **横スクロール対応**: `table-responsive` で、日数が多くてもスマホで横スクロールできます。
- **未提出の強調**: 行に `table-danger`（赤）を付け、名前の横に「未提出」バッジ。0章の要件「未提出は赤く強調」を実現します。
- **セルの中身**: 出勤可なら `開始–終了`、出勤不可（休み）なら `×`、入力なしは空欄。
- **履歴へのリンク**: 提出済みの行から `request_history` へ飛べます（次節で実装）。

## 6-3. 変更履歴のビュー

5章で保存ごとに積んできた `ShiftRequestHistory` を表示します。閲覧できるのは **リーダー、または本人** だけにします。

ファイル: `shifts/views.py`

```python
@login_required
def request_history(request, pk):
    """変更履歴の閲覧。リーダー、または本人のみ。"""
    req = get_object_or_404(ShiftRequest.objects.select_related("user", "period"), pk=pk)
    if not request.user.is_leader and req.user_id != request.user.id:
        messages.error(request, "この履歴を閲覧する権限がありません。")
        return redirect("home")

    histories = list(req.histories.all())  # version_no 降順（モデルのorderingで指定済み）
    return render(
        request,
        "shifts/history.html",
        {"req": req, "histories": histories},
    )
```

**解説**

- **権限チェック**: `@leader_required` ではなく `@login_required` にして、ビュー内で「リーダー or 本人」を判定します。クルーは**自分の履歴だけ**見られるようにするためです。
- `req.user_id != request.user.id` … 他人の提出履歴を覗けないようにするガード。
- `select_related("user", "period")` … 表示に使う関連（提出者・期間）を1クエリで先読み。
- `req.histories.all()` … 2章の `related_name="histories"` で逆参照。`ordering = ["-version_no"]` のおかげで新しい版が上に来ます。

> ⚠️ URLの `pk` を書き換えれば他人の履歴URLを作れてしまいます。だからこそ「本人かリーダーか」をサーバー側で必ず確認します。**画面に出していない＝安全、ではない**点が重要です。

URLも足します。

ファイル: `shifts/urls.py`（`urlpatterns` に追記）

```python
    path("requests/<int:pk>/history/", views.request_history, name="request_history"),
```

## 6-4. 変更履歴のテンプレート

ファイル: `shifts/templates/shifts/history.html`（新規作成）

```html
{% extends "base.html" %}

{% block title %}変更履歴 | {{ req }}{% endblock %}

{% block content %}
<div class="d-flex justify-content-between align-items-center mb-2">
  <h1 class="h4 mb-0">変更履歴</h1>
  <a class="btn btn-outline-secondary btn-sm" href="{% url 'home' %}">← ホーム</a>
</div>
<p class="text-muted small mb-3">
  {{ req.user.name|default:req.user.username }} ／ {{ req.period }}
</p>

{% if not histories %}
  <div class="alert alert-secondary">履歴がありません。</div>
{% endif %}

{% for h in histories %}
<div class="card shadow-sm mb-3">
  <div class="card-header d-flex justify-content-between">
    <span>バージョン {{ h.version_no }}</span>
    <span class="text-muted small">{{ h.created_at|date:"Y/m/d H:i" }}</span>
  </div>
  <div class="card-body">
    <table class="table table-sm mb-2">
      <tbody>
        {% for d in h.snapshot.days %}
        <tr>
          <td class="text-nowrap">{{ d.work_date }}</td>
          <td>
            {% if d.is_available %}{{ d.start_time }}–{{ d.end_time }}{% else %}<span class="text-muted">×（不可）</span>{% endif %}
          </td>
        </tr>
        {% endfor %}
      </tbody>
    </table>
    {% if h.snapshot.note %}<div class="small text-muted">備考: {{ h.snapshot.note }}</div>{% endif %}
  </div>
</div>
{% endfor %}
{% endblock %}
```

**解説**

- **JSONをそのまま展開**: `h.snapshot` は5章で保存したJSON（辞書）。テンプレートで `h.snapshot.days` のように**辞書のキーをドットで**たどれます。
- バージョンごとにカードを作り、その時点の各日と備考を表示します。これで「いつ・どう変えたか」を追えます。

## 6-5. 動作確認

### ✅ 一覧と履歴を確認

1. 複数のクルーで何件か提出しておく（1人は未提出のまま残す）。
2. **リーダーでログイン** → ホームの期間カードで「提出状況を見る」。
3. 提出済みは時刻、休みは `×`、未提出の行は**赤＋「未提出」バッジ**で表示される。「提出: 2 / 3 名」のような件数も合っている。
4. 提出済みの行の「変更履歴」をクリック → バージョン順に内容が並ぶ。
5. **クルーでログインし直し**、提出画面の「変更履歴を見る」から自分の履歴は見られる。一方、他人の履歴URL（`/requests/<別人のID>/history/`）を直接開くと **ホームへ戻され**「権限がありません」と出る。

---

## 6-6. つまずきポイント

- **一覧が重い／クエリが多い** → `prefetch_related("days")` を付け忘れ。先読みでN+1を防ぐ。
- **クルーが表に出ない** → ロールが `staff` になっていない、または `is_active=False`。/admin で確認。
- **`snapshot.days` が表示されない** → 保存時にJSON構造が違う。5章の `_save_submission` のキー名（`days` / `work_date` 等）と一致しているか確認。
- **他人の履歴が見えてしまう** → ビューの「本人 or リーダー」判定を入れ忘れ。

---

## この章のまとめ

- 「クルー × 各日」の表を組み立て、未提出を赤く強調した。
- `prefetch_related` / 辞書引きでN+1を避けた。
- 履歴は `@login_required` ＋ ビュー内判定で「本人 or リーダー」に限定した。
- JSONスナップショットをテンプレートでそのまま展開して履歴を表示した。
