# 講義資料の評価と発展編ロードマップ

このドキュメントは、今回作成・編集した講義資料（`docs/lecture/` の **10〜13章・間章・終章**、および開発編/本番編への再編）を評価し、改善点と発展編の提案をまとめたものです。

**評価の観点**

- **A. 分かりやすさ／イメージ**：開発から保守運用までの流れが、初めての人でも理解でき、全体像をつかめるか。
- **B. 再現性**：読者が手を動かして「今できているものと同じもの」を作れるか。

> 0〜9章は今回の作成対象外（既存）のため、内容そのものの深い評価は控え、全体構成への寄与の観点でのみ言及します。

> ⚠️ **本ドキュメントの「総評」〜「対応状況」は、章の再編・本番編の拡充より前の評価**で、章番号も旧体系（例：当時の「12章＝本番運用」は現在の13章）です。**本番編をつまずき予防型に書き換えた後の最新評価は、末尾の「[再評価]（本番編つまずき予防対応後）」を参照してください。**

## 総評

「**作る → 守る（Git/テスト）→ 公開する → 運用する（DB移行）**」という一連の流れを、実際のコマンドと出力・トラブル対処まで含めて体験できる構成になっており、観点A（流れの分かりやすさ）は高水準です。一方、観点B（再現性）は **環境依存値のハードコード**と**前提環境の暗黙化**により、「別のPC・別のユーザー名」での再現に手当てが必要です。下記の改善で大きく底上げできます。

---

## よかった点

### A. 分かりやすさ／イメージ

- **段階的な物語構成**：開発編（動くアプリ）→ 間章（振り返り）→ 本番編（公開）→ 終章（総括）と、習得の流れが一本道。初学者が「今どこにいるか」を見失いにくい。
- **「なぜ」を都度説明**：単なる手順書でなく理由を添えている。特に以下は理解を促す良い設計。
  - Gunicorn単体だと静的ファイルが404 →「だからNginxが要る」と動機づけ（12章）。
  - `dumpdata` で `contenttypes`/`auth.permission` を除外する理由（13章）。
  - 開発/本番で構成が違う理由を1項目ずつ表で対比（終章）。
- **実際のトラブルを教材化**：静的ファイルの **403（権限）** を実際に踏んで `chmod o+x` で解決する流れ（12章）は、現場で必ず出会う種類のつまずきで価値が高い。
- **安全第一の運用思想**：MySQL移行で「無停止準備 → 短いメンテ → いつでもロールバック」を明示（13章）。"戻せる経路を最初に用意する"という実務の鉄則が伝わる。
- **記法の一貫性**：⚠️（注意）🔰（補足）✅（動作確認）が全章で統一。各章末に「動作確認」と「つまずきポイント」があり自己点検しやすい。

### B. 再現性

- **コマンドが具体的で、期待出力つき**：`200`／`Active: active (running)`／`Installed N object(s)` など「こう出れば成功」が明記され、判定できる。
- **検証手段が充実**：`curl` のステータス確認、`/proc/<pid>/environ` での実プロセス確認、誤パスワードログインでのDB接続確認など、"本当にできたか"を多角的に確かめている。
- **成果物がリポジトリにある**：`deploy/` に Nginx 設定・systemd ユニット・手順READMEが実在し、コピーして使える。
- **非破壊・可逆**：DB切替は既定SQLiteを維持し、ロールバック手順も提示。安心して試せる。

## 特に改善が必要な点

優先度順（高＝再現性に直結）。

### 🔴 高：再現性のための環境依存値と前提の明示

1. **パス・ユーザー名・サービス名がハードコード**
   - 12章本文・`deploy/` に `/home/nagin/ShiftManager`・`User=nagin`・`server_name localhost`・サービス名 `shiftmanager` が直書き（12章だけで `/home/nagin` が9箇所）。読者は自分の環境に合わせて置換が必須だが、**「どこを何に変えるか」の一覧がない**。
   - **改善案**：本番編の冒頭に「置換する値の早見表」を置く。例：

     | プレースホルダ | 意味 | あなたの値の例 |
     |---|---|---|
     | `<PROJECT_DIR>` | アプリの絶対パス | `/home/あなた/ShiftManager` |
     | `<OSUSER>` | 実行ユーザー | `whoami` の結果 |
     | `<SERVER_NAME>` | 公開ホスト名 | `localhost` / `shift.example.com` |

     以降は `<PROJECT_DIR>` 等で記述し、具体値は一箇所で管理する。

2. **前提環境が暗黙**
   - 本番編は **Ubuntu/Debian系（`apt`）＋ systemd ＋ WSL/Linux** を前提にしているが、章の必須条件として明記されていない。Mac や Windows ネイティブの読者は `apt`／`systemd` の時点で詰まる。
   - **改善案**：本番編の最初に「動作確認環境：Ubuntu 24.04 / WSL2、`systemd` 有効、`apt` 利用」を明記し、他OSは「WSL もしくは Linux VM を推奨」と誘導。

3. **MySQL 認証まわりのOS差**
   - MySQL 8 の `caching_sha2_password`／root の `auth_socket` は環境で挙動が分かれやすい。`Access denied`（`@localhost` ソケット vs `@127.0.0.1` TCP）は触れているが、認証プラグイン起因のハマりは未記載。
   - **改善案**：13章のつまずきに「`caching_sha2_password` 関連でドライバが古いと失敗 → `mysqlclient` を最新に」を追記。

### 🟡 中：構成の整合と完結性

4. **章番号と命名の不整合**：9章タイトルが「**発展編1**」のままだが、対になる「発展編2」は本番運用へ改称済みで存在しない。再編後の体系（開発編/本番編）と齟齬。
   - **改善案**：9章を「9章 権限と提出ルール（発展機能）」等へ改称し、"発展編N" の連番をやめる。

5. **間章/終章にファイル番号がない**：`間章_開発編のまとめ.md`／`終章_全体のまとめ.md` は数値prefixがなく、フォルダ表示上の並びが崩れる。読む順序が目次依存。
   - **改善案**：`11b_間章…` / `99_終章…` のように整列用prefixを付けるか、READMEに「読む順序は目次に従う」と明記。

6. **HTTPS化が最後まで実演されない**：12章・終章で「HTTPS導入時に」と繰り返し先送りされ、"公開できるアプリ"の絵が **HTTP止まり**で完結しない。
   - **改善案**：発展編で `certbot` による HTTPS 化を1章として実演（下記ロードマップ①）。

### 🟢 低：仕上げ

7. **用語の初出説明が薄い**：WSGI／リバースプロキシ／ソケット vs TCP／`auth_socket` などが軽い説明のみ。巻末に**用語ミニ集**があると親切。
8. **依存バージョン固定が緩い**：`requirements.txt` が `>=` 指定。完全再現には `pip freeze > requirements.lock` での固定版も案内したい。
9. **視覚資料がない**：CLI主体なので許容範囲だが、GitHubのリポジトリ作成画面や `systemctl status` の出力例など、**スクリーンショット**が数枚あると初学者の不安が減る。

## 発展編ロードマップ（余力がある人向け）

「作って公開・運用できる」の次に、**実務水準へ引き上げる**テーマを、取り組む順に提案します。難易度は ★（やさしい）〜★★★（要調査）。各テーマは独立しているので興味のある所から着手可。

### ① HTTPS化でインターネット公開を完成させる ★★
- **目的**：本編の積み残し。暗号化通信で安全に外部公開する。
- **学べること**：`certbot`（Let's Encrypt）での証明書取得、Nginxの443設定と80→443リダイレクト、`SECURE_SSL_REDIRECT`/`SESSION_COOKIE_SECURE`/`CSRF_COOKIE_SECURE`/`CSRF_TRUSTED_ORIGINS`。`check --deploy` の警告が消える達成感つき。
- **なぜ先頭**：本編が HTTP 止まりの弱点（改善点6）を直接解消する。

### ② 設定と秘密情報の管理を洗練する ★★
- **目的**：環境変数の散在を整理し、設定を安全・体系的に。
- **学べること**：`django-environ` や `.env`（python-dotenv）、settings の base/dev/prod 分割、秘密情報をリポジトリから完全分離する運用。
- **なぜ**：本番編で増えた `DJANGO_*`/`DB_*` を一元管理でき、再現性（改善点1）の整備とも相性が良い。

### ③ CIで自動テストを回す（GitHub Actions）★★
- **目的**：pushのたびに11章のテストを自動実行し、壊れたら気づく。
- **学べること**：`.github/workflows` の書き方、CI上でのDB（SQLite/MySQLサービスコンテナ）、テスト失敗でマージを止める発想。
- **なぜ**：10章（Git）・11章（テスト）の自然な延長。品質を"仕組み"で守る。

### ④ Docker / docker-compose 化 ★★★
- **目的**：app（Gunicorn）+ Nginx + MySQL を1コマンドで再現。
- **学べること**：Dockerfile、`docker-compose.yml`、コンテナ間ネットワーク、ボリュームでのデータ永続化。
- **なぜ**：**環境依存のハードコード問題（改善点1・2）を根本から解消**でき、「誰のPCでも同じものが動く」を実現する。再現性を最重視するなら本命。

### ⑤ 定期処理とリマインド通知 ★★★
- **目的**：締切の自動クローズや、未提出者への締切前メール。
- **学べること**：カスタム management command、`cron`/systemd timer での定期実行、Djangoのメール送信（SMTP）。`profile` で集めたメールを活用。
- **なぜ**：「運用」を一歩進め、無人でも回るシステムにする。

### ⑥ バックアップ自動化とログ・監視 ★★
- **目的**：障害に備える運用基盤。
- **学べること**：`mysqldump` + `cron` + 世代ローテーション、`media/` の定期退避、Djangoの `LOGGING` 設定、（任意で）Sentryでのエラー通知。
- **なぜ**：本番編で触れた運用メモを、実際に"自動で回る"形にする。

### ⑦ 機能・UX拡張 ★★〜★★★
- **目的**：使い勝手と適用範囲を広げる。
- **候補**：提出状況のCSV/PDF出力、カレンダーUI・スマホ最適化（必要なら HTMX）、複数店舗対応（マルチテナント）、DRFでのAPI化＋モバイル連携。
- **なぜ**：実利用のフィードバックを反映する「育てるフェーズ」の練習。

> **おすすめの進め方**：まず①でHTTPSを完成させ、②③で運用と品質の足場を固め、再現性を重視するなら④へ。⑤〜⑦は興味に応じて。①〜③は本編の延長で着手しやすく、④以降は新しい概念が増えるぶん学びも大きいテーマです。
