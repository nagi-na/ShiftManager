# ShiftManager デプロイ・運用マニュアル

新しいサーバーへ ShiftManager を設置する手順と、設置後のサーバー操作（起動・停止・再起動・リロード・更新・バックアップ）をまとめたものです。コピーして使えるよう、コマンドを順番に並べています。

- 記法：`$` で始まる行はコマンド（`$` は入力しない）。`#` はコメント。
- 🔰 補足 / ⚠️ 注意 / ✅ 確認。

---

## 0. はじめに（前提と方針）

- **対象環境**：Ubuntu 22.04/24.04 などの Debian 系 Linux（`apt` と `systemd` が使えること）。`ps -p 1 -o comm=` が `systemd` を返せばOK。
- **開発機とは別のサーバー**に置くことを前提とします（[12章の環境分離](lecture/12_開発編のまとめ.md)）。**本番に開発用の練習データ・設定を持ち込まない**こと。
- **DB は MySQL（本番向け）** を主に説明し、小規模なら SQLite でも可（4章で分岐）。
- 秘密情報（`SECRET_KEY`・DBパスワード）は**Gitに入れず**、サーバー上の root 保護ファイルにだけ置きます。
- **本番の動かし方は2通り**：①**直接インストール**（systemd＋Nginx＋MySQL ＝ **第1〜3部**）／②**Docker compose**（**第4部**・[19章](lecture/19_Docker化.md)）。**どちらか一方**を選びます（第1〜3部は①が前提）。

### この手順で使う値（自分の環境に置き換える）

| 記号 | 意味 | 例 |
| --- | --- | --- |
| `<APP_USER>` | アプリを動かすOSユーザー（専用ユーザー推奨） | `shift` |
| `<APP_DIR>` | アプリの設置先（Webサーバーが読める場所） | `/srv/shiftmanager` |
| `<HOST>` | 公開に使うホスト名/IP | `shift.example.com` / `203.0.113.10` |
| `<DBPASS>` | MySQLアプリユーザーのパスワード | （強いランダム文字列） |

> 🔰 アプリは `/home/ユーザー` 直下より **`/srv` 等のWebサーバーが読める場所**に置くのがおすすめ（`/home` は権限で弾かれやすい）。以下は `<APP_DIR>=/srv/shiftmanager` を例にします。

# 第1部 新規サーバーへのセットアップ（初回だけ）

> 全体の流れ：**パッケージ導入 → コード取得 → venv → DB → 管理者 → 静的ファイル → Gunicorn(systemd) → Nginx →（公開ならHTTPS）→ 確認**

## 1. OSパッケージを入れる

```
$ sudo apt-get update
$ sudo apt-get install -y python3-venv python3-pip git nginx
# MySQL を使う場合は追加で
$ sudo apt-get install -y mysql-server default-libmysqlclient-dev build-essential pkg-config
$ sudo systemctl enable --now mysql      # MySQLを起動＋自動起動
```

## 2. コードを取得する

```
$ sudo mkdir -p /srv && sudo chown $USER:$USER /srv
$ cd /srv
$ git clone https://github.com/nagi-na/ShiftManager.git shiftmanager
$ cd /srv/shiftmanager
```

> 🔰 専用ユーザーで運用する場合は、`<APP_DIR>` の所有者をそのユーザーにします（例：`sudo chown -R shift:shift /srv/shiftmanager`）。以降のコマンドはアプリを動かすユーザーで実行します。

## 3. 仮想環境と依存パッケージ

```
$ python3 -m venv venv
$ ./venv/bin/python -m pip install --upgrade pip
$ ./venv/bin/pip install -r requirements.txt
# MySQLを使うのに mysqlclient が入っていなければ
$ ./venv/bin/pip install mysqlclient
$ ./venv/bin/python -c "import django; print('Django', django.get_version())"   # ✅ バージョン表示
```

## 4. データベースを用意する

開発の SQLite は持ち込まず、**本番DBは空から**作ります。

### 4-A. MySQL（推奨）

```
# DBと接続ユーザーを作成（<DBPASS> は強いパスワードに置換）
$ sudo mysql -e "CREATE DATABASE IF NOT EXISTS shift_manager CHARACTER SET utf8mb4;
  CREATE USER IF NOT EXISTS 'shift'@'localhost' IDENTIFIED BY '<DBPASS>';
  GRANT ALL PRIVILEGES ON shift_manager.* TO 'shift'@'localhost';
  FLUSH PRIVILEGES;"

# テーブル作成（マイグレーション）
$ DJANGO_DB=mysql DB_PASSWORD='<DBPASS>' ./venv/bin/python manage.py migrate
```

### 4-B. SQLite（小規模なら可）

```
$ ./venv/bin/python manage.py migrate     # BASE_DIR/db.sqlite3 が作られる
```

> ⚠️ SQLite は Django 5.1 が **3.31 以上**を要求します。古い環境では `pysqlite3-binary` の併用が必要になる場合があります。本番で同時アクセスが見込まれるなら MySQL を選んでください。

## 5. 管理者アカウントを作成

```
# MySQLの場合（環境変数を付ける）
$ DJANGO_DB=mysql DB_PASSWORD='<DBPASS>' ./venv/bin/python manage.py createsuperuser
# SQLiteの場合
$ ./venv/bin/python manage.py createsuperuser
```

以降、クルー等の実アカウントは**アプリの管理画面（S6）**から作成します。

## 6. 静的ファイルを集める

```
$ DJANGO_DEBUG=0 DJANGO_ALLOWED_HOSTS=<HOST> \
    ./venv/bin/python manage.py collectstatic --noinput     # staticfiles/ に集約
```

## 7. Gunicorn を systemd サービスとして常駐させる

秘密情報を含む**サービス定義ファイルを作成**します（root保護）。`SECRET_KEY` は **英数字だけ**で生成します（記号入りだと systemd に弾かれ、ワーカーごとに鍵が変わって「勝手にログアウト」する原因になります）。

```
# 英数字64文字の固定SECRET_KEYを生成（値は画面に出さずシェル変数へ）
$ SECRET=$(python3 -c "import secrets,string; print(''.join(secrets.choice(string.ascii_letters+string.digits) for _ in range(64)))")

# サービス定義を作成（MySQLの例。SQLiteなら DJANGO_DB / DB_PASSWORD 行を削る）
$ sudo tee /etc/systemd/system/shiftmanager.service >/dev/null <<EOF
[Unit]
Description=ShiftManager Gunicorn daemon
After=network.target

[Service]
User=$USER
Group=$USER
WorkingDirectory=/srv/shiftmanager
Environment=DJANGO_DEBUG=0
Environment=DJANGO_ALLOWED_HOSTS=<HOST>,localhost,127.0.0.1
Environment=DJANGO_SECRET_KEY=$SECRET
Environment=DJANGO_DB=mysql
Environment=DB_PASSWORD=<DBPASS>
ExecStart=/srv/shiftmanager/venv/bin/gunicorn shift_manager.wsgi:application --bind 127.0.0.1:8000 --workers 3
Restart=on-failure

[Install]
WantedBy=multi-user.target
EOF

$ sudo chmod 640 /etc/systemd/system/shiftmanager.service   # 秘密情報を含むので保護
$ sudo systemctl daemon-reload
$ sudo systemctl enable --now shiftmanager                  # 起動＋自動起動
$ systemctl is-active shiftmanager                          # ✅ active
```

> ⚠️ `<HOST>` は**実際にアクセスするホスト名**を必ず入れる（`DEBUG=False` では `ALLOWED_HOSTS` に無いと `400 DisallowedHost`）。
> 🔰 全ワーカーが同じ鍵を持つか確認：`for p in $(pgrep -f "gunicorn shift_manager"); do tr '\0' '\n' </proc/$p/environ | grep -c '^DJANGO_SECRET_KEY='; done` → すべて `1`。

## 8. Nginx を前段に置く（静的配信＋中継）

```
$ sudo tee /etc/nginx/sites-available/shiftmanager >/dev/null <<'EOF'
server {
    listen 80;
    server_name <HOST>;
    client_max_body_size 12m;

    location /static/ { alias /srv/shiftmanager/staticfiles/; access_log off; expires 7d; }
    location /media/  { alias /srv/shiftmanager/media/; }
    location / {
        proxy_pass http://127.0.0.1:8000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
EOF

# 上の <HOST> を実際の値に書き換えてから：
$ sudo sed -i 's/<HOST>/あなたのホスト名/' /etc/nginx/sites-available/shiftmanager
$ sudo ln -sf /etc/nginx/sites-available/shiftmanager /etc/nginx/sites-enabled/shiftmanager
$ sudo rm -f /etc/nginx/sites-enabled/default
$ sudo nginx -t                       # ✅ syntax is ok / test is successful
$ sudo systemctl reload nginx
```

> ⚠️ アプリが `/home/ユーザー` 配下にあると Nginx(`www-data`) が静的ファイルに到達できず **403** になることがあります。`/srv` 等に置くか、経路に通過権限を与えてください。

## 9. （インターネット公開する場合）HTTPS

公開ドメインがあるなら **Cloudflare Tunnel** が手軽です（ポート開放不要・正規HTTPS）。手順は [16章](lecture/16_Cloudflareトンネルで公開.md)・[17章](lecture/17_名前付きトンネルと運用のつまずき.md)。HTTPS一本化するときは、サービス定義に次を足して `daemon-reload`＋`restart`：

```
Environment=DJANGO_BEHIND_TLS_PROXY=1
Environment=DJANGO_CSRF_TRUSTED_ORIGINS=https://<HOST>
Environment=DJANGO_SECURE_COOKIES=1
```

> ⚠️ `DJANGO_SECURE_COOKIES=1` を入れると Cookie が HTTPS 限定になり、HTTP では login が維持できません。**HTTPS一本化が前提**のときだけ有効化します。

## 10. 動作確認

```
$ curl -s -o /dev/null -w "%{http_code}\n" http://localhost/login/                    # 200
$ curl -s -o /dev/null -w "%{http_code}\n" http://localhost/static/admin/css/base.css # 200（Nginxが配信）
```

ブラウザで `http(s)://<HOST>/` を開き、ログインできれば完了です。

# 第2部 サーバーの操作（日常運用）

ShiftManager は3つ（公開時は4つ）のサービスで動きます。操作はすべて `systemctl`。

| サービス | 役割 |
| --- | --- |
| `shiftmanager` | アプリ本体（Gunicorn） |
| `nginx` | 前段Webサーバー（静的配信・中継） |
| `mysql` | データベース（MySQL利用時） |
| `cloudflared` | 公開トンネル（Cloudflare利用時） |

## 操作の早見表（アプリ＝shiftmanager の例）

| やりたいこと | コマンド |
| --- | --- |
| **起動** | `sudo systemctl start shiftmanager` |
| **停止** | `sudo systemctl stop shiftmanager` |
| **再起動**（停止→起動。コード/設定反映） | `sudo systemctl restart shiftmanager` |
| **状態を見る** | `systemctl status shiftmanager` |
| **稼働中か一言で** | `systemctl is-active shiftmanager` |
| **自動起動ON/OFF** | `sudo systemctl enable shiftmanager` / `disable` |
| **完全に無効化（mask）** | `sudo systemctl mask shiftmanager`（`start` も依存起動も不可になる） |
| **mask を解除（unmask）** | `sudo systemctl unmask shiftmanager` |
| **ログを追う**（Ctrl+Cで終了） | `sudo journalctl -u shiftmanager -f` |
| **直近のログ** | `sudo journalctl -u shiftmanager -n 100 --no-pager` |

`nginx` / `mysql` / `cloudflared` も同じ形（`shiftmanager` を置き換え）。

> 🔰 **`stop` / `disable` / `mask` の違い**（紛らわしいので整理）
> - **`stop`**：今すぐ止めるだけ。次回の手動 `start` も、`enable` 済みならブート時起動も**できる**。
> - **`disable`**：**自動起動（ブート時）をやめる**だけ。手動 `start` は**できる**。
> - **`mask`**：**完全に無効化**。サービス定義を `/dev/null` にリンクし、`start` も他サービスからの**依存起動もできなくする**最も強い停止。`disable` より強力。解除は **`unmask`**。
>
> 使いどころ：メンテで「絶対に動かしたくない」、別の仕組みに置き換えて旧サービスを封じたい等。
> ```
> $ sudo systemctl stop shiftmanager && sudo systemctl mask shiftmanager     # 確実に止めて封印
> $ sudo systemctl unmask shiftmanager && sudo systemctl start shiftmanager  # 封印解除して再開
> ```
> ⚠️ `mask` したまま `start` しようとすると `Unit is masked.` で失敗します。動かないときは `systemctl status` で `masked` 表示を確認 → `unmask`。

### Nginx の「リロード」と「再起動」の違い

```
$ sudo nginx -t && sudo systemctl reload nginx   # リロード：設定を無停止で再読込（推奨）
$ sudo systemctl restart nginx                   # 再起動：一瞬接続が切れる
```

> 🔰 Nginx は設定変更時、まず `nginx -t` で文法チェック → `reload`。`reload` なら接続を保ったまま反映できます。

## コードを更新したとき（GitHubの変更を反映）

```
$ cd /srv/shiftmanager
$ git pull                                                   # 最新コードを取得
$ ./venv/bin/pip install -r requirements.txt                 # 依存が増えていれば
$ DJANGO_DB=mysql DB_PASSWORD='<DBPASS>' ./venv/bin/python manage.py migrate          # モデル変更があれば
$ DJANGO_DEBUG=0 DJANGO_ALLOWED_HOSTS=<HOST> ./venv/bin/python manage.py collectstatic --noinput   # 静的変更があれば
$ sudo systemctl restart shiftmanager                        # アプリ再起動で反映
```

> 🔰 `migrate`/`collectstatic` を手動で打つときは環境変数（`DJANGO_DB` 等）を付けます。systemd 経由の起動ではサービス定義の `Environment=` が使われるので不要です。

## 設定（環境変数・サービス定義）を変えたとき

```
$ sudo nano /etc/systemd/system/shiftmanager.service   # 値を編集
$ sudo systemctl daemon-reload                          # 定義の再読込（必須）
$ sudo systemctl restart shiftmanager                   # 反映
# 反映確認（秘密値は出さない）
$ systemctl show shiftmanager -p Environment | tr ' ' '\n' | grep -E 'DJANGO_|DB_' | grep -v SECRET_KEY
```

> ⚠️ サービス定義を編集したら **必ず `daemon-reload`**。これを忘れると古い定義のままです。

## バックアップと復元

```
# MySQL バックアップ（日付つき）
$ mysqldump -u shift -p shift_manager > backup_$(date +%F).sql
# 復元
$ mysql -u shift -p shift_manager < backup_2026-06-08.sql

# SQLite の場合はファイルをコピー
$ cp /srv/shiftmanager/db.sqlite3 backup_$(date +%F).sqlite3

# アップロードPDF（media/）も併せて退避
$ tar czf media_$(date +%F).tgz -C /srv/shiftmanager media
```

> 🔰 定期バックアップは `cron` で自動化できます（例：毎日 `mysqldump`）。バックアップファイルもサーバー外へ控えておくと安心です。

## サービス一式の起動/停止（メンテ時）

```
# まとめて停止（依存の逆順で）
$ sudo systemctl stop shiftmanager nginx
# まとめて起動
$ sudo systemctl start mysql shiftmanager nginx
```

# 第3部 困ったとき（クイック診断）

| 症状 | 最初に見るところ |
| --- | --- |
| **502 / 「Bad Gateway」** | `shiftmanager` が落ちている。`systemctl status shiftmanager`、`journalctl -u shiftmanager -n 50` |
| **400 (DisallowedHost)** | `DJANGO_ALLOWED_HOSTS` にアクセス先のホストが無い。サービス定義を修正→`daemon-reload`→`restart` |
| **ログインで403 (CSRF)** | HTTPS公開時に `DJANGO_CSRF_TRUSTED_ORIGINS` と `DJANGO_BEHIND_TLS_PROXY=1` を設定（第1部9） |
| **何度もログアウト** | `SECRET_KEY` がワーカー間で不一致。英数字の固定鍵をサービス定義に（第1部7） |
| **CSS/管理画面が崩れる** | `collectstatic` 未実行、または Nginx `/static/` の `alias` パス違い |
| **静的が403** | アプリの配置場所の権限（`/home`配下は要注意）。`/srv`等へ |
| **全ページ500** | `DB_PASSWORD` 誤り／MySQL未起動（`systemctl status mysql`） |
| **設定を変えたのに反映されない** | `daemon-reload`（systemd）／`reload`（nginx）忘れ |
| **`start` しても起動しない（`Unit is masked.`）** | `mask` 状態（`systemctl status` が `masked`）→ `sudo systemctl unmask <名前>` してから `start` |

ログの見方：
```
$ sudo journalctl -u shiftmanager -f      # アプリ（Gunicorn/Django）のログ
$ sudo tail -f /var/log/nginx/error.log   # Nginx のエラーログ
```

公開前の総点検：
```
$ DJANGO_DEBUG=0 DJANGO_ALLOWED_HOSTS=<HOST> DJANGO_SECRET_KEY=dummy \
    ./venv/bin/python manage.py check --deploy
```

# 第4部 Docker compose で運用する場合（別方式）

Docker で動かすなら、第1〜2部の `apt install`・systemd・Nginx/MySQL の個別設定は**不要**です。代わりに `compose.yaml` 一式（[19章](lecture/19_Docker化.md) で用意）で操作します。

## 初回セットアップ（要点）

```
$ cd /srv/shiftmanager           # リポジトリを置いた場所
$ cp .env.example .env           # 値を自分のものに（SECRET_KEYは英数字、DB_PASSWORD 等）
$ docker compose up -d --build   # ビルドして起動
$ docker compose exec web python manage.py migrate         # テーブル作成
$ docker compose exec web python manage.py createsuperuser # 管理者作成
```

ブラウザで `http://<ホスト>:${HTTP_PORT:-80}/` を開いて確認。詳細・各ファイルの中身は [19章](lecture/19_Docker化.md)。

## 操作早見表（Docker版）

| やりたいこと | コマンド |
| --- | --- |
| **起動**（ビルド込み） | `docker compose up -d --build` |
| **停止**（コンテナ削除・**データは残る**） | `docker compose down` |
| **状態** | `docker compose ps` |
| **ログ** | `docker compose logs -f web`（`nginx`/`db` も） |
| **再起動** | `docker compose restart web` |
| **コンテナ内でコマンド** | `docker compose exec web python manage.py migrate` |
| **コード更新を反映** | `git pull && docker compose up -d --build`（必要なら `exec web … migrate`） |
| **静的ファイル更新** | `docker compose run --rm web python manage.py collectstatic --noinput` |
| **完全削除（ボリュームも！）** | `docker compose down -v` ⚠️ **DBの中身ごと消える** |

## 「直接インストール」との操作対応

| 直接インストール（第2部） | Docker（compose） |
| --- | --- |
| `sudo systemctl start/stop/restart shiftmanager` | `docker compose up -d` / `down` / `restart web` |
| `systemctl status` / `journalctl -u shiftmanager -f` | `docker compose ps` / `docker compose logs -f web` |
| `sudo systemctl reload nginx`（設定変更時） | `docker compose restart nginx` |
| `migrate`・`collectstatic`（venvで実行） | `docker compose exec web python manage.py …` |
| 秘密情報＝systemd drop-in（640） | 秘密情報＝`.env`（Git外） |

> 🔰 `systemctl mask` に相当するのは Docker では「そのサービスを起動しない／コンテナを止めて作らない」です（`docker compose stop <名前>` で個別停止、`up` 時に対象を外す等）。

## バックアップ（Docker）

```
# DBダンプ（dbコンテナ内の mysqldump を使う。パスワードはコンテナのenvを参照）
$ docker compose exec -T db sh -c 'exec mysqldump -ushift -p"$MYSQL_PASSWORD" shift_manager' > backup_$(date +%F).sql

# media ボリュームを丸ごと退避（ボリューム名は docker volume ls で確認。例: shiftmanager_media）
$ docker run --rm -v shiftmanager_media:/m -v "$PWD":/out alpine tar czf /out/media_$(date +%F).tgz -C /m .
```

復元：`docker compose exec -T db sh -c 'exec mysql -ushift -p"$MYSQL_PASSWORD" shift_manager' < backup_YYYY-MM-DD.sql`

## 注意

- **`docker compose down -v` の `-v`** はボリューム（DB・media）まで消す。通常の停止は `down`（`-v` なし）。
- **静的ファイルは名前付きボリューム**にあるため、コード/静的を変えた後は上表の `collectstatic` で更新（`up --build` だけでは古いまま残ることがある）。
- **HTTPS公開**は cloudflared を別途（ホスト側 or もう1コンテナ）で前段に置く（[16-17章](lecture/16_Cloudflareトンネルで公開.md)）。
